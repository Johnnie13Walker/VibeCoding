# prompt-master: runner

## Назначение

Этот файл — практический способ вызвать `prompt-master` прямо сейчас, до того как появится runtime-узел Cloudbot. Идея простая: вставить системный промпт в начало диалога с LLM, дать вход — получить структурированный orchestration brief.

Файл нужен потому, что `system-prompt.md`, `routing-rules.md` и другие артефакты описывают правила, но не описывают, как именно их применить в обычном чате с Claude / GPT / Codex.

## Когда использовать

- Любая нетривиальная задача (multi-step, multi-agent, требующая аудита).
- Любая визуальная задача (графики, таблицы, дашборды, презентации, фронтенд).
- Любая задача, где есть риск для production, данных пользователя или интеграций.
- Любая хаотичная мысль, голосовая расшифровка, поток идей — всё, что нужно сначала нормализовать.

Когда **не** использовать:

- Однострочные правки (опечатка, переименование одной переменной) — идти сразу к `Мастеру` с конкретным file ownership.
- Чисто справочные вопросы («что делает эта функция») — идти к `Следопыту` или просто к LLM.

## Способ 1. Запуск в чате с LLM

### Шаг 1. Вставить системный промпт

В начало нового диалога вставить полный текст из [system-prompt.md](system-prompt.md). LLM теперь играет роль `prompt-master`.

Дополнительно желательно приложить:

- [routing-rules.md](routing-rules.md) — чтобы маршрутизация была обоснована;
- [prompt-templates.md](prompt-templates.md) — чтобы prompts для агентов были в утверждённой форме;
- [memory-rules.md](memory-rules.md) — если задача связана с domain `personal` / `belberry` / `shared`;
- релевантный `MEMORY.md` (`personal/MEMORY.md`, `belberry/MEMORY.md` или `shared/MEMORY.md`) — чтобы память домена учитывалась.

### Шаг 2. Сформулировать вход

Вход может быть любым: хаотичный текст, голосовая расшифровка, эмоциональный поток, короткая идея, неполное ТЗ. Чистить заранее не нужно — `prompt-master` это сам сделает.

Минимальная подсказка LLM, чтобы запуск был стабильным:

```text
Вход:
[ваш хаотичный текст]

Сделай intake по правилам system-prompt.md. Верни ответ в утверждённом формате (простой задачи или multi-agent).
```

### Шаг 3. Получить результат

LLM должна вернуть один из двух форматов из system-prompt.md:

- **простая задача** — 7 секций, заканчивающихся `Production-ready prompt` и `Критерии качества`;
- **multi-agent** — 10 секций, включающих `Orchestration flow`, `Agent routing`, `Prompts для агентов`, `Context handoff`, `Acceptance checklist`.

И один из 4 статусов: `ОК` / `Нужны данные` / `Есть риски` / `Не готово`.

### Шаг 4. Запустить агентов

Полученные prompts можно запускать:

- последовательно в новых чатах с LLM, вручную копируя prompt и контекст;
- передавать `Мастеру` как задание с явным file ownership (когда нужен write);
- держать в одном диалоге, играя ролями агентов по очереди — для быстрых задач.

После каждой стадии возвращать результат `prompt-master`-чату и просить его обновить статус и acceptance checklist.

## Способ 2. Запуск в Claude Code / Codex / Cursor

Эти инструменты уже подхватывают `AGENTS.md` как контекст автоматически. Чтобы `prompt-master` активно использовался:

1. Убедиться, что в корне проекта есть `AGENTS.md` (есть).
2. В первом сообщении сессии явно попросить: «Перед началом работы используй `prompt-master` из `agents/prompt-master/system-prompt.md` для intake. Верни orchestration brief, прежде чем что-либо делать».
3. Только после получения brief давать разрешение на исполнение.

Это снимает с инструмента соблазн сразу прыгнуть в реализацию без нормализации входа.

## Способ 3. Будущий runtime (заглушка)

Когда появится runtime-узел Cloudbot, последовательность будет такой:

1. Сообщение пользователя приходит в Cloudbot `orchestrator` (Telegram или другой канал).
2. `orchestrator` маршрутизирует это в LLM-вызов с `system-prompt.md` как system message.
3. Ответ парсится по `schemas/agent-routing.schema.json`.
4. Каждый агент из `agents` массива запускается отдельным LLM-вызовом со своим prompt.
5. Результаты собираются и проверяются `Испытателем`.
6. Финальный артефакт + статус возвращается пользователю.

До появления этого runtime используются Способы 1 и 2.

## Минимальный шаблон ввода

Для удобства — можно копировать прямо в чат:

```text
Ты `prompt-master`. Системный контекст: agents/prompt-master/system-prompt.md.
Учти: routing-rules.md, prompt-templates.md, memory-rules.md, quality-checklist.md.

Domain: [personal | belberry | shared | unknown]

Вход:
[твоя задача / мысль / поток / неполное ТЗ]

Сделай intake. Если задача простая — выдай простую форму ответа. Если multi-agent — выдай полную форму со всеми 10 секциями. В конце укажи статус: ОК / Нужны данные / Есть риски / Не готово.
```

## Acceptance для самого runner

Раннер считается рабочим, если:

- из любой хаотичной формулировки `prompt-master` возвращает структурированный brief с указанным статусом;
- prompts для агентов содержат все 11 обязательных полей из `system-prompt.md`;
- для визуальных задач brief всегда включает связку `Продуктолог + Дизайнер` до реализации и `Испытатель + Дизайнер` после;
- для write-задач явно указан file ownership;
- ни одна задача не отдана пользователю без статуса аудита.

## Ограничения

- LLM может «забывать» правила длинного system-prompt в долгом диалоге — лучше открывать новый чат для каждого крупного intake.
- Без runtime каждый handoff между агентами делается вручную — это быстро становится утомительным; лучше использовать раннер для intake, а саму реализацию вести в Claude Code / Codex.
- Память (`MEMORY.md`) приходится прикладывать руками — runtime будет это делать автоматически.
