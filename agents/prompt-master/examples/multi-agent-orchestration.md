# Пример: multi-agent orchestration

## Задача

Создать HTML-дашборд по маркетинговым каналам, который показывает эффективность лидогенерации, стоимость заявки, качество лидов и рекомендации по перераспределению бюджета.

## Orchestration flow

```text
Параллельно:
- Продуктолог: KPI и управленческий сценарий
- Дизайнер: визуальная структура и требования к читаемости
- Справочник: ограничения источников данных/API

Затем:
- analytics-agent: расчёты и модель данных
- Маркетолог: нарратив для руководителя и команды
- Креативщик: концепция отчёта, если стандартный dashboard слишком плоский
- Мастер: реализация только в конкретных файлах
- Испытатель: проверка данных, сценариев, логов
- Дизайнер: финальный визуальный аудит
```

## Prompt для Продуктолога

```markdown
Ты `Продуктолог`. Нужно определить, какое решение руководитель должен принять по HTML-дашборду маркетинговых каналов.

Контекст:
Дашборд показывает эффективность лидогенерации, стоимость заявки, качество лидов и рекомендации по перераспределению бюджета.

Сформируй:
- главный управленческий вопрос;
- 5-7 ключевых KPI;
- default view;
- фильтры;
- порядок блоков;
- какие метрики убрать, чтобы не перегрузить отчёт.

Формат ответа:
- решение пользователя;
- KPI;
- структура экрана;
- критерии полезности.
```

## Prompt для Дизайнера

```markdown
Ты `Дизайнер`. Нужно спроектировать читаемый и профессиональный HTML-дашборд по маркетинговым каналам.

Проверь:
- визуальную иерархию;
- плотность данных;
- таблицы и графики;
- подписи;
- цвета;
- состояния пустых данных;
- читаемость на desktop и mobile.

Формат ответа:
- layout-рекомендации;
- графики и таблицы;
- цветовые правила;
- visual QA checklist.
```

## Prompt для Мастера

```markdown
Ты `Мастер`, единственный write-worker.

File ownership:
- reports/marketing_dashboard.html
- reports/marketing_dashboard.css
- tests/marketing_dashboard.test.ts

Задача:
Реализовать HTML-дашборд по утверждённой структуре Продуктолога и Дизайнера.

Ограничения:
- менять только файлы из file ownership;
- не трогать соседние файлы;
- не добавлять новые зависимости без согласования;
- сохранить существующий стиль проекта.

Проверки:
- запустить тесты;
- открыть dashboard локально;
- проверить mobile и desktop;
- проверить пустые состояния.

Формат ответа:
- изменённые файлы;
- что реализовано;
- результаты проверок;
- оставшиеся риски.
```

## Acceptance checklist

- [ ] Главный вывод понятен за 30 секунд.
- [ ] Есть рекомендации по бюджету.
- [ ] Метрики не конфликтуют друг с другом.
- [ ] Данные проверены.
- [ ] Dashboard читается на mobile и desktop.
- [ ] Финальный аудит Дизайнера пройден.
