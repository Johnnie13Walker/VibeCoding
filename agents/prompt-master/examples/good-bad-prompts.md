# Good/bad prompts

## Плохой prompt

```text
Сделай нормальный отчёт по продажам, чтобы было красиво и понятно.
```

Проблемы:

- нет бизнес-цели;
- нет аудитории;
- нет источника данных;
- нет критериев качества;
- непонятен формат результата;
- слово "красиво" не превращено в проверяемые требования.

## Хороший prompt

```markdown
## Контекст

Нужен ежедневный отчёт по продажам для РОПа. Данные берутся из CRM. Отчёт должен показывать, где теряются лиды, кто из менеджеров требует внимания и какие действия нужно сделать сегодня.

## Цель

Помочь РОПу за 2 минуты принять управленческие решения по менеджерам и сделкам.

## Задача

Спроектировать структуру отчёта, KPI, фильтры и визуальную форму. Отчёт должен быть пригоден для автоматической отправки в 09:30 МСК.

## Ограничения

- не больше 7 основных KPI;
- не использовать метрики без источника данных;
- отдельно показать риски false positive при малом числе лидов;
- визуальный результат должен пройти аудит Дизайнера.

## Ожидаемый результат

- структура отчёта;
- список KPI;
- правила подсветки менеджеров;
- блок "что делать сегодня";
- acceptance checklist.

## Критерии качества

- главный вывод понятен за 30 секунд;
- есть конкретные действия;
- отчёт не перегружен;
- данные проверяемы;
- подходит для mobile и desktop.
```

## Плохой orchestration prompt

```text
Пусть агенты посмотрят и сделают как надо.
```

Проблемы:

- не указаны агенты;
- нет порядка выполнения;
- нет handoff;
- нет критериев приёмки;
- нет владельца реализации;
- нет финальной проверки.

## Хороший orchestration prompt

```markdown
## Orchestration flow

1. `Следопыт`: найти файлы отчёта, источники данных и тесты.
2. `Продуктолог`: определить KPI и управленческий сценарий.
3. `Дизайнер`: определить визуальную структуру.
4. `Мастер`: изменить только `reports/sales_dashboard.html` и `tests/sales_dashboard.test.ts`.
5. `Испытатель`: запустить тесты, проверить mobile/desktop, проверить логи.
6. `Дизайнер`: провести финальный визуальный аудит.

## Context handoff

Каждый агент передаёт:
- что выяснил;
- какие ограничения обнаружил;
- какие файлы/данные использовал;
- какие риски остались.

## Acceptance checklist

- тесты проходят;
- dashboard открывается;
- нет визуального мусора;
- цифры совпадают с источником;
- есть финальный статус аудита.
```
