# Пример: хаотичная задача → production-ready prompt

## Хаотичный вход

```text
слушай надо сделать норм отчет по продажам а то там опять какая-то фигня в таблице менеджеры не понимают кто норм работает кто тупит и где лиды теряются, хочу чтобы утром это само приходило и было понятно кого хвалить кого спасать кого пинать
```

## Нормализованная задача

Нужно спроектировать ежедневный управленческий отчёт по продажам, который показывает эффективность менеджеров, потери по этапам воронки и конкретные действия для РОПа: кого хвалить, кого поддержать, где есть риск провала и какие лиды требуют внимания. Отчёт должен автоматически приходить утром.

## Классификация

- продажи
- аналитика
- автоматизация
- визуальный артефакт
- управленческий отчёт

## Недостающие данные

- источник CRM-данных;
- список этапов воронки;
- список менеджеров;
- расписание отправки;
- канал доставки: Telegram, email, Google Sheets или HTML;
- правила оценки менеджеров;
- период анализа.

## Orchestration flow

```text
Продуктолог → Дизайнер → analytics-agent → lev-petrovich → Мастер/automation-agent → Испытатель → Дизайнер
```

## Production-ready prompt

```markdown
## Контекст

Нужно создать ежедневный управленческий отчёт по продажам для РОПа. Отчёт должен не просто показывать данные, а помогать принять решение: кого хвалить, кого спасать, где теряются лиды и какие сделки требуют внимания.

## Цель

Сформировать автоматический утренний отчёт, который показывает эффективность менеджеров, зоны риска и конкретные управленческие действия.

## Задача

Спроектировать структуру отчёта, KPI, визуальную форму и orchestration flow для автоматической доставки отчёта утром.

## Агенты

1. `Продуктолог`: определить решения, KPI, default view и порядок блоков.
2. `Дизайнер`: определить таблицы, графики, плотность, цвета, подписи и риски восприятия.
3. `analytics-agent`: описать расчёты и структуру данных.
4. `lev-petrovich`: проверить управленческую полезность для продаж.
5. `automation-agent` или `Мастер`: реализовать автоматизацию после утверждения file ownership.
6. `Испытатель`: проверить данные, расписание, доставку и логи.
7. `Дизайнер`: финальный визуальный аудит.

## Ожидаемый результат

- структура отчёта;
- список KPI;
- схема данных;
- расписание доставки;
- prompts для агентов;
- acceptance checklist.

## Критерии качества

- РОП понимает за 2 минуты, где проблема;
- есть список конкретных действий;
- нет перегруза метриками;
- данные проверяемы;
- отчёт готов к автоматической отправке;
- визуальная структура читается на мобильном и desktop.

## Риски

- неправильная классификация этапов CRM;
- демотивация менеджеров из-за некорректного ранжирования;
- ложные выводы при малом числе лидов;
- сбой отправки утром.

## Формат ответа

Верни orchestration brief и prompts для каждого агента.
```
