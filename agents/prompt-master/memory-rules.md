# prompt-master: memory rules

## Назначение памяти

`prompt-master` должен учитывать память workspace, чтобы не дублировать существующие решения и правильно выбирать контур работы.

## Источники памяти

Текущие известные источники:

- `personal/MEMORY.md` — экспериментальный контур, обучение, быстрые прототипы.
- `belberry/MEMORY.md` — business/production контур, клиенты, микросервисы, безопасность, масштабирование.
- будущие `MEMORY.md` внутри новых доменов workspace.

## Правила чтения памяти

Перед генерацией orchestration flow проверь:

1. Есть ли релевантный `MEMORY.md` в домене задачи.
2. Есть ли уже принятые архитектурные решения.
3. Есть ли ограничения по безопасности, production или стилю работы.
4. Есть ли существующие агенты или workflows, которые нельзя дублировать.

## Выбор домена

Если задача про эксперименты, обучение, быстрый прототип или личный AI-контур, используй `personal`.

Если задача про клиентов, production, бизнес-процессы, продажи, отчёты, безопасность или масштабирование, используй `belberry`.

Если задача общая и должна быть доступна всем контурам, используй корневой `agents/` или `shared/`.

## Memory handoff

В prompt для агента добавляй краткий memory context:

```markdown
## Memory context

Домен:

Релевантные решения:

Ограничения:

Что не дублировать:
```

## Запреты

- Не переписывать `MEMORY.md` без явного запроса.
- Не считать память абсолютной истиной, если она конфликтует с текущим заданием пользователя.
- Не переносить production-ограничения в personal-эксперименты без причины.
- Не переносить experimental-подходы в production-контур без ревизии.

## Обновление памяти

Предлагать обновить память, если:

- принято новое архитектурное решение;
- добавлен новый агент;
- появился новый workflow;
- изменился routing matrix;
- обнаружено правило, которое должно сохраняться между задачами.
