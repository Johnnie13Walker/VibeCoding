# prompt-master: routing rules

## Принцип маршрутизации

Маршрутизация строится по четырём признакам:

1. Домен задачи.
2. Тип результата.
3. Риск ошибки.
4. Нужна ли реализация или только подготовка prompt/orchestration.

`prompt-master` не исполняет предметные задачи. Он выбирает агентов, формирует prompts и orchestration flow.

## Правила выбора Мастера

`Мастер` подключается только если:

- задача требует изменения файлов;
- известен точный file ownership;
- есть ограниченный набор файлов или модулей;
- запрещены изменения соседних файлов без отдельного разрешения;
- есть понятные критерии проверки.

Пример корректного назначения:

```text
Мастер отвечает только за:
- agents/prompt-master/system-prompt.md
- agents/prompt-master/routing-rules.md

Не менять остальные файлы. Не откатывать чужие изменения.
```

## Правила подключения Справочника

`Справочник` обязателен, если:

- пользователь просит актуальную информацию;
- задача зависит от API/SDK/версии библиотеки;
- есть риск изменения правил платформы;
- используются внешние ограничения: Google, Telegram, OpenAI, Bitrix24, WHOOP, SEO-поисковики, GitHub Actions.

## Правила подключения Ревизора

`Ревизор` обязателен, если:

- результат влияет на production;
- есть риск false positive/false negative;
- есть финансовые или клиентские последствия;
- есть интеграции с внешними API;
- меняются контракты данных;
- задача включает автоматизацию, которая может отправлять сообщения или менять данные.

## Правила подключения Испытателя

`Испытатель` обязателен после:

- изменения кода;
- изменения workflow;
- изменения визуального артефакта;
- изменения интеграции;
- изменения расписания, cron или deploy-логики;
- генерации отчёта, который будет показан пользователю или команде.

## Правила визуального контура

Если задача визуальная, маршрут всегда содержит:

1. `Продуктолог`
2. `Дизайнер`
3. исполнитель
4. `Испытатель`
5. `Дизайнер`

Исключений нет, если результат будет показан человеку.
