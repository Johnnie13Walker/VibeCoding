# prompt-master: workflows

## Workflow 1. Быстрая нормализация задачи

Использовать, когда пользователь дал короткую мысль или простую задачу.

Шаги:

1. Прочитать вход.
2. Убрать шум, эмоции и повторы.
3. Сформулировать нормализованную задачу одним абзацем.
4. Определить домен.
5. Выбрать одного основного агента.
6. Сформировать production-ready prompt.
7. Добавить критерии качества.

Выход: один prompt для одного агента.

## Workflow 2. Multi-agent orchestration

Использовать, когда задача требует нескольких компетенций, этапов или независимой проверки.

Шаги:

1. Определить бизнес-цель.
2. Определить конечный артефакт.
3. Разбить задачу на этапы.
4. Выявить зависимости между этапами.
5. Выбрать агентов.
6. Определить порядок выполнения.
7. Сформировать prompt для каждого агента.
8. Описать context handoff между агентами.
9. Назначить финальную проверку.
10. Сформировать acceptance checklist.

Выход: orchestration flow и набор prompts.

## Workflow 3. Визуальный артефакт

Использовать для графиков, таблиц, Excel, Google Sheets, дашбордов, HTML-отчётов, презентаций, диаграмм и фронтенд-экранов.

Обязательный порядок:

1. `Продуктолог`: определить решение, которое должен принять человек.
2. `Дизайнер`: определить визуальную структуру, иерархию и критерии читаемости.
3. Профильный исполнитель: реализовать артефакт.
4. `Испытатель`: проверить фактическую готовность и критичные сценарии.
5. `Дизайнер`: провести финальный визуальный аудит.
6. `prompt-master`: собрать итоговый статус и финальный prompt/brief для пользователя.

Выход: готовый к исполнению orchestration brief.

## Workflow 4. Разработка

Использовать для кода, архитектуры, интеграций, DevOps и автоматизации.

Порядок:

1. `Следопыт`: разведка по структуре, файлам, тестам, рискам минимального пути.
2. `Справочник`: официальные docs/API/версии, если задача зависит от внешних правил.
3. `Ревизор`: проверка рисков до реализации.
4. `Мастер` или `dev-agent`: реализация только при конкретном file ownership.
5. `Испытатель`: тесты, ручные сценарии, логи.
6. `Ревизор`: повторная проверка для high-risk изменений.

Выход: scoped engineering execution plan и prompts по ролям.

## Workflow 5. Research и источники

Использовать, когда важны актуальные данные, правила платформы, документация, API или ограничения сервиса.

Порядок:

1. Сформулировать research question.
2. Назначить `Справочник`.
3. Запросить только проверяемые источники.
4. Потребовать ссылки и даты доступа.
5. Запретить неподтверждённые выводы.
6. Передать ограничения профильному агенту.

Выход: source-backed brief.

## Workflow 6. Бизнес-отчёт или управленческая аналитика

Использовать для отчётов по продажам, финансам, маркетингу, CRM, performance и KPI.

Порядок:

1. `Продуктолог`: решение, KPI, default view.
2. `analytics-agent`: расчёты и структура данных.
3. `Дизайнер`: визуальная форма.
4. `Маркетолог`: нарратив для аудитории, если отчёт будет показан людям.
5. `Испытатель`: проверка чисел, фильтров, edge cases.

Выход: prompt на создание отчёта и prompt на аудит.
