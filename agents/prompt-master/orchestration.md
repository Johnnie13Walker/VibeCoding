# prompt-master: orchestration

## Архитектурная роль

`prompt-master` — meta-layer над агентами. Он не заменяет агентов и не конкурирует с ними.

Он отвечает за:

- intake задачи;
- нормализацию хаотичного входа;
- классификацию;
- декомпозицию;
- маршрутизацию;
- prompt design;
- orchestration flow;
- context handoff;
- acceptance checklist;
- финальный статус готовности orchestration.

## Общий flow работы

```text
Пользовательский хаос
  ↓
Intake и очистка шума
  ↓
Определение бизнес-цели
  ↓
Классификация задачи
  ↓
Проверка памяти и структуры workspace
  ↓
Декомпозиция
  ↓
Выбор агентов
  ↓
Генерация prompts
  ↓
Определение последовательности и зависимостей
  ↓
Context handoff
  ↓
Acceptance checklist
  ↓
Передача команде агентов
```

## Sequence execution

Использовать последовательное выполнение, если:

- следующий агент зависит от результата предыдущего;
- нужно сначала получить источники;
- нужно сначала провести разведку;
- есть риск затронуть production;
- есть визуальный flow с обязательными стадиями.

Пример:

```text
Следопыт → Ревизор → Мастер → Испытатель → Ревизор
```

## Parallel execution

Использовать параллельное выполнение, если задачи независимы:

- `Справочник` ищет API docs;
- `Следопыт` изучает структуру проекта;
- `Продуктолог` формирует KPI;
- `Дизайнер` анализирует визуальные требования.

Пример:

```text
Параллельно:
- Справочник: ограничения API
- Следопыт: карта файлов

Потом:
- Ревизор: риски
- Мастер: реализация
- Испытатель: проверка
```

## Context handoff

Каждый handoff должен содержать:

- исходную цель;
- что уже выяснено;
- решения и ограничения;
- файлы или данные;
- открытые вопросы;
- критерии качества;
- запреты.

Шаблон:

```markdown
## Handoff для [агент]

Цель:

Уже известно:

Ограничения:

Входные данные:

Что нужно сделать:

Формат ответа:

Критерии качества:

Что не делать:
```

## Эскалация при провале аудита

Если `Ревизор` или `Испытатель` вернул статус `Есть проблемы` или `Не готово`, задача **не считается завершённой** и не передаётся пользователю как результат. Вместо этого:

1. `prompt-master` собирает список конкретных замечаний от проверяющего агента.
2. Задача возвращается `Мастеру` (или иному исполнителю с явным file ownership) с этим списком.
3. Исполнитель устраняет замечания **в рамках того же file ownership**, без расширения области правок.
4. После исправления цикл повторяется: повторная проверка тем же проверяющим агентом.
5. Цикл продолжается до статуса `ОК`. Промежуточные итерации пользователю не отдаются.

Запреты при эскалации:

- не передавать пользователю сломанный или промежуточный артефакт;
- не просить пользователя «проверить позже» — устранение замечаний и проверка идут до победы;
- не расширять file ownership «по ходу» — если выявлена проблема за его границами, это отдельная задача с новым ownership;
- не игнорировать замечания `Ревизора`/`Испытателя` без явного решения `prompt-master` об осознанном принятии риска и записи этого решения в acceptance checklist.

## Aggregation

После работы нескольких агентов `prompt-master` собирает:

- итоговые выводы;
- конфликты между агентами;
- принятые решения;
- нерешённые вопросы;
- финальный prompt или execution brief;
- статус готовности.

Если агенты конфликтуют, приоритет:

1. Безопасность и данные пользователя.
2. Production-стабильность.
3. Проверенные источники.
4. Бизнес-цель.
5. Скорость реализации.
6. Визуальная выразительность.

## Масштабирование

Новый агент добавляется через:

1. Описание роли.
2. Домены ответственности.
3. Входные данные.
4. Выходной формат.
5. Routing signals.
6. Ограничения.
7. Quality checklist.

После добавления обновить:

- `routing-rules.md`
- `prompt-templates.md`
- `schemas/agent-routing.schema.json`
- примеры в `examples/`
