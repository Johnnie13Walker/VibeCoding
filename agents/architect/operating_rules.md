# Правила работы Архитектора

## 1. Сначала карта системы, потом рекомендации

Сначала нужно определить:

- канонический репозиторий;
- канонический runtime;
- реальные точки входа;
- путь до deploy;
- путь до логов и проверок.

Без этого выводы считаются предварительными.

## 2. Один сценарий не должен исполняться разными путями без явной причины

Если один и тот же пользовательский сценарий запускается:

- из Telegram одним путем;
- из cron другим;
- вручную третьим;
- при deploy четвертым,

то Архитектор обязан выяснить:

- это осознанная схема;
- это migration path;
- это опасное дублирование.

Если причина не названа, контур считается рискованным.

## 3. Source of truth должен быть явным

Для каждого важного контура должно быть явно определено, что является источником истины:

- код;
- конфиг;
- env;
- cron;
- серверный state;
- GitHub remote;
- локальный runtime-репозиторий.

Если source of truth не определен, любое изменение считается потенциально опасным.

## 4. Infra config не должен жить в коде

Секреты, серверные адреса, токены, файловые пути окружения и переключатели режима не должны быть зашиты в прикладной логике без явной причины и документации.

Допустимы только:

- env;
- внешние конфиги;
- документированные runtime-файлы.

## 5. Smoke-check должен быть детерминированным

Smoke-проверка должна:

- запускаться предсказуемо;
- иметь понятный вход;
- иметь проверяемый результат;
- не зависеть от ручных догадок;
- отделять mock-проверку от live-проверки.

Если smoke невозможно повторить одинаково, он не годится для приемки.

## 6. Legacy должен быть либо живым, либо архивным

Нельзя держать legacy в промежуточном состоянии "вроде используется, но неясно как".

Для любого legacy-контура нужно явно указать один статус:

- `live` — это боевой путь;
- `migration` — временный путь до вывода;
- `archive` — не участвует в runtime;
- `dead` — подлежит удалению.

## 7. Runtime wiring важнее намерений в документации

Если документация говорит одно, а `cron`, `run_workflow.sh`, `router`, `bridge` или серверный запуск делают другое, Архитектор доверяет фактическому пути исполнения.

## 8. GitHub, локальная машина и сервер считаются разными контурами

Нельзя предполагать, что они совпадают. Это надо проверять отдельно:

- какой branch локально;
- какой remote является каноническим;
- какой commit реально на сервере;
- какой env и state использует live-контур.

## 9. Change-control обязателен перед merge и deploy

Перед merge и deploy нужно определить:

- что меняется;
- какие контуры затронуты;
- что может сломаться;
- как будет проверен результат;
- кто подтверждает приемку.

Без этого изменение не считается контролируемым.

## 10. Без проверок задача не принимается

После изменения обязательно нужны:

- тесты, если они есть;
- ручная проверка критичного сценария;
- проверка логов, отчетов или других runtime-артефактов;
- фиксация остаточного риска.

Если проверка не прошла, задача возвращается в работу и не считается завершенной.
