## ЗАЧЕМ

Авто-отчёт структурно уже близок к ручному эталону, но **мельче по содержанию**. Эталон для сверки:
`belberry/sales/daily_report/отчеты/Сводка_продаж_2026-05-29.html`.

Задача — поднять отчёт до эталонной глубины **и** добавить новую аналитику, которой нет ни у кого (история в БД уже копится). Согласованный с заказчиком scope — ниже, бери его целиком, **ничего сверх**.

Ключевая ценность продукта (не нарушать): руководитель и РОП за 30 секунд видят честную картину вчерашнего дня. Всё новое работает на эту цель.

## ЖЁСТКИЕ ПРАВИЛА (стоп-условия)

1. **Scope-fence.** Делаешь РОВНО коммиты 1–9 ниже. Любое «заодно отрефакторю/добавлю» — СТОП, в финальный отчёт пунктом «предложения», не в код.
2. **Не деплоить, не пушить в прод, не трогать VPS/cron/nginx.** Прод-интеграции — отдельный заход.
3. **Не выдумывать данные.** Если поля нет в payload/БД — выводить «нет данных» или опускать блок. Никаких синтетических цифр/цитат.
4. **Только ОП + телемаркетинг** во всех блоках (фильтр уже есть, не ослаблять).
5. **Русский** во всём (код-комментарии, коммиты, тексты отчёта). Без англицизмов в текстах отчёта.
6. **Атомарные коммиты** по списку, Conventional Commits, trailer `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`. Bisect-friendly.
7. **Тесты зелёные после каждого коммита.** `pytest` в `runner/` (сейчас 102 green) + web-тесты не ломать. **Не коммитить при red.**
8. **Данные, требующие новых вызовов Bitrix (продукт сделки, ниша), НЕ тянуть в этот заход.** Если поля уже нет в payload — помечать «направление: нет данных» и фиксировать в выходе как будущее. Это граница scope, не обходить.
9. **Детерминированные расчёты — в Python (transform/payload), НЕ в LLM.** LLM отвечает за формулировки и разбор, числа считает код. Health-score, Δ, ТМ-воронка, автотег риска, петля обещаний — код.
10. При сомнении в формулировке/пороге — НЕ угадывать молча: реализовать разумный дефолт и вынести вопрос в блок «ОТКРЫТЫЕ ВОПРОСЫ» финального выхода.

**Цель:** обогатить схему разбора встречи (данные) под эталонную глубину.
**Файлы:** `runner/src/analyze_llm.py` (схема разбора + `analyze_meeting`), `runner/tests/`.
**Сделать:** расширить JSON-схему разбора встречи новыми полями (LLM заполняет из транскрипта):
