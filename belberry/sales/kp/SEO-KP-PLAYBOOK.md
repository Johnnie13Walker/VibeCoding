# SEO-КП Belberry — плейбук шаблона

Как собрать продающий КП по SEO под нового клиента: data-driven слайды
автозаполняются аудитом из API, остальное правится по брифу/сделке.

**Эталон:** `clients/med-shushary/` (16 слайдов) — копируем его, как gutaclinic
для общего движка (см. [README.md](README.md)).

---

## Движок аудита — `build_kp.py`

Тянет ПРОВЕРЯЕМЫЕ SEO-данные через **официальный API PR-CY** (тот же сервис, что
за `analitics.belberry-dev.ru`). Ключ — в `~/.config/vibecoding/assistant/secrets/prcy.env`.

```bash
python3 build_kp.py <домен> [конкурент1 конкурент2 ...]
# пример:
python3 build_kp.py med-shushary.ru flmed.ru feniksmed.ru cl-veda.ru
```

Один GET-вызов на домен (`pr-cy.ru/api/v1.1.0/analysis/base/`) отдаёт всё; лимит 2 req/сек.

**На выходе:**
| Файл | Что это | Куда вставить |
|---|---|---|
| `audit.json` | полный аудит клиента + конкурентов | — |
| `seo_benchmark.html` | `<tbody>` сравнения ИКС/индексации | слайд «SEO-показатели: вы и соседи» |
| `problem_solution.html` | `<tr>` «проблема → решение» из реальных флагов | слайд «Что мешает → как чиним» |

**Данные, которые собирает (всё — настоящее, не оценки):**
ИКС (офиц. индекс Яндекса), индексация Яндекс/Google, доноры, PR-CY Rank, доля
органики, отказы, время на сайте, HTTPS/robots/sitemap/Schema, скорость загрузки,
**динамика трафика помесячно** (пик → текущее → %, для слайда «остановить просадку»).

## Метрика-факты — `metrika_audit.py`

Превращает «оценку» в «факт», если счётчик клиента доступен в агентском аккаунте
Belberry (311 счётчиков) или выдан гостевой доступ. Токен — в `metrika-belberry.env`.

```bash
python3 metrika_audit.py <counter_id | домен> [дней=90]
python3 metrika_audit.py panaceadoc.ru 90
```

Отдаёт РЕАЛЬНЫЕ: визиты/посетители, **долю органики** (Метрика честнее PR-CY:
panaceadoc 49% против вранья PR-CY 12%), топ органических поисковых фраз, **цели и
конверсии** (например, отправка формы 1.65%) → слайд «Прогноз» из оценки в факт.
Если счётчика нет — честно сообщает «нужен гостевой доступ». Выход: `metrika.json`.

> **Контроль достоверности:** запусти на клиенте с готовым пресейл-аудитом —
> ИКС/индексация должны совпасть. На `med-shushary.ru` = ИКС 80, Я 354, G 130. ✓

**Чего движок НЕ делает** (остаётся сеошнику / правится вручную из брифа):
глубокая семантика и позиции по запросам (платные тулы), контентные проблемы
(карточки врачей, тексты услуг), стратегия линкбилдинга.

## Bitrix-материалы — `bitrix_audit.py`

По ID сделки → весь смысловой блок. Перед запуском синкнуть state
(`bash shared/scripts/bitrix-sync-state.sh`).

```bash
python3 bitrix_audit.py 12866
```

Тянет: ядро сделки (домен, выручка компании, услуги), **бриф СП1056 с подписями
полей** (читаемые ответы: услуги, ЛПР, ЦА, УТП, **конкуренты прямо из брифа**),
транскрипт встречи СП1048, переписку Wazzup → `bitrix.json`. Это сырьё для
потребностей / среднего чека / списка конкурентов (его можно сразу скормить `build_kp.py`).

## Локальный аудит — `prodoctorov_audit.py`

```bash
python3 prodoctorov_audit.py <url_prodoctorov> [url2 ...]
```

С карточки ПроДокторов: число врачей, отзывов, рейтинг (Schema.org) → `prodoctorov.json`.
URL карточки — из поиска (ручной шаг).

**Какие площадки скриптуются (проверено 10.06):**
- ✅ **ПроДокторов** — обычный GET (200, полный HTML) → в движке.
- ❌ **Zoon** — 429 / Cloudflare-анти-бот.
- ❌ **НаПоправку** — JS-challenge ServicePipe (отдаёт заглушку-спиннер, не контент).
- ❌ **Яндекс.Карты** — 302 / JS; официальный Maps API рейтинг/отзывы не отдаёт (см. ниже).

Zoon / НаПоправку / Карты в скриптовый движок не добавляем. Их рейтинг/отзывы при
сборке конкретного КП берём **через WebFetch в сессии** (он рендерит — НаПоправку
отдал 4.5/382) или headless-браузером, если когда-то понадобится массово. ПроДокторов
для медицины репутацию закрывает.

официально с Карт не вытащить → закрываем через **ПроДокторов** (рейтинг/врачи/отзывы есть).

## Как собрать новый SEO-КП

```bash
cd belberry/sales/kp

# 1. Копируем эталон
cp -r clients/med-shushary clients/<домен>

# 2. Собираем аудит (клиент + 3–5 конкурентов по локации/нише)
python3 build_kp.py <домен> <конк1> <конк2> <конк3>
#   → audit.json, seo_benchmark.html, problem_solution.html

# 3. Поднимаем данные сделки из Bitrix (услуги, средний чек, ЛПР, транскрипт)
#    шаблон вызова — см. shared/scripts + memory reference_bitrix24_access

# 4. Правим clients/<домен>/kp.html:
#    - слайд 7  ← вставить seo_benchmark.html в <tbody>
#    - слайд 4  ← вставить problem_solution.html + добавить контент-строки из брифа
#    - слайд 12 ← цели ИКС: текущий → max(ИКС соседей) из audit.json
#    - слайд 6  ← конкуренты: реальные сайты/адреса/фишки (НЕ выдумывать цифры)
#    - титул/состав/стоимость/календарь ← цены (115 000 + НДС 5%, подарок-внедрение)
#    - прогноз ← цифры продакшна, средний чек из брифа

# 5. Сборка PDF + PPTX
bash build.sh clients/<домен> "КП Belberry — <домен>"

# 6. Открыть PDF, проверить что ничего не обрезано
```

## Правила (из этого кейса)

- **Цены из сметы без НДС → × 1,05 (НДС 5% медицина) → подарок вместо скидки**
  (внедрение SEO-правок бесплатно при старте до дедлайна).
- **Цифры конкурентов — только из API/наблюдаемых фактов.** Трафик/заявки соседей
  снаружи не измеряются — НЕ приводим. ИКС/индексация — можно (это реальные данные).
- **Контроль достоверности на слайде:** показываем, что наш ИКС/страницы совпали с
  пресейл-аудитом — значит и цифры соседей настоящие.
- **Структурные claim'ы про сайт клиента — только после краула, НЕ из булевых флагов
  аудита.** `crawl_commercial` даёт `doctors=✓/✗` по главной — это НЕ значит, что на
  сайте есть раздел врачей или что услуг мало. Перед слайдом «Что мешает» проверь
  руками: `curl` главной → собрать ссылки `/directions/`, `/uslugi/`, `/vrachi/`;
  пробить `/vrachi/`,`/specialisty/`,`/komanda/` на 404; глянуть гео в `<title>` услуг.
  **Осторожно с URL-транслитерацией и угадыванием путей:** русские ЧПУ часто через
  нестандартную схему (ц→cz, щ→shh, й→j), а карточки сущностей живут на отдельном
  префиксе. Не делай вывод «страницы нет» по одному угаданному URL — бери ссылки из
  меню главной и из карточки списка (на med-shushary список врачей — `/speczialisty/`,
  а каждая карточка — `/doctor/<slug>/`).
  **Антипример med-shushary 10.06 — ЧЕТЫРЕ неверных claim'а подряд, прежде чем дошёл до
  правды** (пользователь поймал на дилетантстве): (1) «мало страниц под услуги» — у них
  26+ направлений; (2) «нет раздела врачей / 404» — есть на `/speczialisty/`; (3)
  «карточки плоские, нет стажа/образования/Schema» — у каждого врача отдельная страница
  `/doctor/<slug>/` с образованием, опытом («18+»), фото и онлайн-записью; (4) «путь
  Услуга→Врач оборван» — страница услуги линкует врачей направления.
  **Реальный (проверенный) недобор оказался узким:** нет МЕДИЦИНСКОЙ Schema-разметки
  (`Physician` на врачах, `MedicalProcedure` на услугах — стоит только базовая
  BreadcrumbList/Organization) → нет расширенных карточек в выдаче. Сам сайт сделан
  добротно, и честный пивот: «проблема не в сайте, а в видимости» (трафик −45%, Google
  130/354, ИКС 80, 1 донор).
  **Правило:** структурный claim про чужой сайт писать ТОЛЬКО открыв конкретную целевую
  страницу (карточку врача/услуги, а не список) и посчитав сигналы grep'ом; типы Schema
  проверять точно (`"@type":"Physician"`), а не по подстроке `schema.org`; не вставлять
  выдуманных процентов («~10% запросов»). Источник помечать «🔍 проверено на сайте <дата>».
- **Кейсы — кликабельны** на конкретные страницы `belberry.net/works/...`.
- **Конверсионные слайды** (сильнее всего двигают продажу): Гарантии/KPI →
  Стоимость бездействия → Belberry vs подрядчик → отзыв из ниши.
- Опыт Belberry — **11 лет (с 2015), 700+ проектов**; Гордиенко — РОП.
