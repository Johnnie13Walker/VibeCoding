# PLAYBOOK — от сделки к коммерческому предложению Belberry

Воспроизводимый конвейер: **сделка в Bitrix24 → потребности клиента → готовое КП**
в фирменном стиле Belberry, опираясь на продуктовую матрицу.

Режим — **с чекпоинтами**: на ключевом шаге показываю выявленные потребности и
предлагаемый набор продуктов, жду подтверждения, потом собираю КП.

## Вход

Один из вариантов от пользователя:
- ID или URL сделки (`https://belberrycrm.bitrix24.ru/crm/deal/details/<ID>/`)
- URL карточки в смарт-процессе (встреча/КП) — тогда поднимаю родителя `parentId2`

## Шаг 1. Сбор данных по сделке (REST Bitrix24)

Скрипт доступа: `shared/scripts/bitrix-sync-state.sh` → токен в
`shared/config/bitrix24-state/install.latest.json` (поле `payload['auth[access_token]']`,
endpoint `payload['auth[client_endpoint]']`).

Поднять:
- **Карточку сделки** (`crm.deal.get`): воронка, стадия, OPPORTUNITY, компания, контакт,
  менеджер, UF-поля (сайт, выручка, ссылки на сметы/КП, причина отвала).
- **Связанные встречи** (SP=1048, `crm.item.list` filter `parentId2`): брифинг, защита КП.
- **Транскрипты встреч** — поле `UF_CRM_16_TRANSCRIPT` (memoai.tech, скачать через
  `crm.controller.item.getFile`). Читать целиком — это первоисточник потребностей.
- **Таймлайн** (`crm.timeline.comment.list`) — переписка, выжимки встреч, ссылки на сметы/КП.
- **Активности** (`crm.activity.list`) — звонки, задачи, история передач сделки.
- **Брифы/КП** — в UF-полях сделки или ссылками в таймлайне (Google Sheets / Drive / Pitch).

> Детали и рабочий пример — в Vault: deal-analysis methodology + кейс gutaclinic-ru.

## Шаг 2. Выявление потребностей (аудит сделки)

Из транскриптов и переписки выписать:
- **Главную задачу клиента** (прямой цитатой). Пример Гута: «лидогенерация, клиентов не хватает».
- **Приоритетные направления/услуги** клиники.
- **Что уже используется** (каналы) и **что не закрыто** (боль).
- **Контекст**: сайт, переезд, конкуренты, средний чек, заявки, ЦА.
- **Кто ЛПР** — не путать контакт сделки с тем, кто принимает решение (на защите КП
  часто появляется новый человек — реальный ЛПР).
- **Прямые запросы клиента** на встрече (что он сам просил: прогноз, тотал, удешевление).

Свериться с CRM-меткой причины/этапа vs реальные слова на звонке.

## Шаг 3. ЧЕКПОИНТ — согласование набора продуктов

Показать пользователю:
1. **Выявленные потребности** (буллеты с цитатами).
2. **Предлагаемый комплекс** — продукты из [[PRODUCT-MATRIX]] под эти потребности,
   с базовыми ценами и обоснованием «почему именно это».
3. **Развилки** (если есть): ORM с ГЕО или без, SEO сейчас или после переноса и т.п.

**Ждать подтверждения / правок.** Не собирать КП до этого.

Маппинг «потребность → продукт» (ориентир):
| Потребность клиента | Продукт из матрицы |
|---|---|
| Мало лидов из поиска | SEO регулярное (+GEO) |
| Переезд на новый сайт | Разовое SEO для переноса |
| Негатив / нет отзывов | ORM (Старт → Развитие → Контроль+) |
| Локальный поток пациентов | ORM + ГЕО (Карты, 2ГИС) |
| Не видят в ChatGPT/Алисе | Нейровыдача (GEO) |
| Нужен быстрый трафик/сезон | Контекст · Гибридный |
| Нет сайта / устарел | Сайт Strapi / Лендинг Tilda + Брендинг |
| Соцсети, контент, прогрев | Программа (Лайт/Базовый/Депозит) |

## Шаг 4. Расчёт цен

Для каждого выбранного продукта:
1. Взять базовый тариф из [[PRODUCT-MATRIX]] (или из сметы сделки, если есть точная).
   В каждой вкладке матрицы есть калькулятор, бриф, шаблон КП и договор по продукту.
2. Откалибровать под объём клиента (часы/площадки/страницы/тип сайта).
3. **НДС зависит от юрлица** (вкладка «НДС»): Акула Тим 22%, НКТ 5%, ИП Егоренко 5%,
   ИП Щемелева 0%. Уточнить, через какое юрлицо идёт сделка. Для Belberry-медицины
   в кейсе Гута — 5%.
4. **Цена без НДС → × (1 + ставка НДС) → × 0,95 (скидка 5% за оплату в 10 дней).**
   Скидку **нельзя** применять к минимальному чеку.
5. Собрать денежный поток: **1-й месяц** (разовое + первый ежемесячный) /
   **2-й и далее** / **всего за 12**. Эта логика — единая на слайдах состава, тотала, календаря.

## Шаг 5. Сборка КП

1. `cp -r clients/gutaclinic clients/<новый>` (см. README.md).
2. Править `kp.html`: титул, состав, детальные слайды продуктов, прогнозы (щадящие +
   дисклеймер), сводный тотал, медиаплан-календарь, дорожная карта, финал с дипломами.
3. Обновить `.pagenum` если поменялось число слайдов.
4. `bash build.sh clients/<новый> "КП Belberry — <клиент>"` → PDF + PPTX.
5. Открыть PDF, проверить что ничего не обрезано.

## Шаг 6. (Опц.) Прикрепить аудит к сделке

Аудит сделки (внутренний) — .docx в поле `UF_CRM_DEAL_AUDIT` через `crm.deal.update`.
**Prod-write делает пользователь/Codex** — Claude готовит файл и команду, не пишет в живую CRM сам.

## Чего НЕ делать

- ❌ Не выдумывать цифры прогнозов там, где их честно нет (нейровыдача — диапазоны).
- ❌ Не давать точные цены без сверки с матрицей/сметчиком.
- ❌ Не путать должности (Гордиенко — РОП).
- ❌ Не собирать КП до чекпоинта согласования набора продуктов.
- ❌ Не писать в прод-CRM напрямую — только готовить команду.
