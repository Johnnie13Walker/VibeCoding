# SPEC: KP-пайплайн MVP (SEO-Belberry, CLI)

> Дата: 2026-06-10. Родитель: [KP-ENGINE-V2-PLAN.md](KP-ENGINE-V2-PLAN.md).
> Объём MVP — один сквозной проход «сделка → данные → заготовка КП» командой,
> без веба и Telegram (это этапы 3–4 плана).

## Цель

Сейлс (или Claude по запросу) запускает ОДНУ команду с ID сделки — и получает
в `clients/<домен>/` собранные данные всех аудитов, заготовку деки из эталона
и чек-лист «что добить руками». Сегодня это 4 ручных запуска + ручное копирование
эталона; antipattern из разбора WON-КП (утечка внутренних листов клиенту)
закрывается тем, что в папке клиента появляются только клиентские артефакты.

## Решения, увязанные с разбором выигравших КП (WON-KP-INSIGHTS.md)

- Смету движок НЕ генерит и цены НЕ выдумывает: цены — зона сметчика (Google Sheets
  по матрице). Пайплайн оставляет в чек-листе пункт «цены из сметы».
- Дека — для SEO/PPC (холодные); для остальных услуг выигрывает постатейная смета —
  шаблон сметы подключим этапом 2.
- Блоков «KPI-гарантии / окупаемость / сроки» в артефактах нет (в выигравших КП их нет).

## CLI

```bash
cd belberry/sales/kp
python3 kp_pipeline.py <deal_id>                # полный проход
python3 kp_pipeline.py <deal_id> --status       # статус стадий
python3 kp_pipeline.py <deal_id> --force audit  # перезапуск стадии
# опции: --client <имя-папки> --competitors d1 d2 --days 90
#        --prodoctorov <url>... --skip-metrika --skip-prodoctorov
```

## Стадии (идемпотентные, состояние в clients/<домен>/kp_job.json)

| # | Стадия | Что делает | Чем | Выход |
|---|---|---|---|---|
| 1 | bitrix | сделка+бриф СП1056+транскрипт+Wazzup | `bitrix_audit.py <deal>` | bitrix.json |
| 2 | audit | PR-CY: ИКС/индексация/трафик/техфлаги + конкуренты (из брифа или --competitors) | `build_kp.py <домен> <конк…>` | audit.json, seo_benchmark.html, problem_solution.html |
| 3 | metrika | факт-визиты/органика/конверсии (если счётчик доступен) | `metrika_audit.py <домен> <дни>` | metrika.json |
| 4 | prodoctorov | рейтинг/врачи/отзывы (если URL передан) | `prodoctorov_audit.py <url…>` | prodoctorov.json |
| 5 | assemble | свод фактов С ИСТОЧНИКАМИ + чек-лист ручных шагов | pure-функция | kp_data.json |
| 6 | scaffold | копия эталона med-shushary в папку клиента, вставка авто-блоков по маркерам (если маркеры есть в kp.html — иначе инструкция) | копирование + inject | clients/<домен>/kp.html |

Сборка PDF/PPTX — как раньше, вручную после правки цен: `bash build.sh clients/<домен> "КП Belberry — <домен>"` (пайплайн печатает готовую команду).

## Контракт kp_data.json (ядро анти-галлюцинации)

```json
{
  "deal_id": 12866, "domain": "med-shushary.ru", "brand": "Belberry",
  "facts": [
    {"key": "iks", "value": 80, "source": "audit.json:pr-cy", "status": "факт"},
    {"key": "visits_month", "value": 8420, "source": "metrika.json", "status": "факт"}
  ],
  "hypotheses": [ {"key": "...", "note": "...", "status": "гипотеза"} ],
  "manual_checklist": [
    "Цены — из сметы сметчика (не выдумывать)",
    "Титул: клиент, дата, дедлайн подарка/скидки",
    "Дисклеймер прогноза (LEGAL-MED-2026 §2: без гарантий результата)",
    "Проверить «имеются противопоказания…» если в деке есть рекламные обещания"
  ]
}
```

Правило: всё, что не имеет source — в `hypotheses` или в чек-лист, в факты не попадает.

## Чего MVP сознательно НЕ делает

- Не пишет в Bitrix (прикреп КП — вручную/этап 3).
- Не генерит текст слайдов LLM-ом (этап 4).
- Не считает цены и НДС.
- Не трогает прод SCC: это локальный CLI рядом с остальными скриптами `kp/`.
  Перенос в `runner/src/` + Postgres `kp_jobs` — при интеграции в SCC (этап 3).

## Тесты

`tests/test_kp_pipeline.py` — pure-функции: разбор домена из сделки, извлечение
конкурентов из текста брифа, планирование стадий (идемпотентность/force),
сборка kp_data (факты только с источниками), вставка авто-блоков по маркерам.
Сетевых вызовов в тестах нет.
