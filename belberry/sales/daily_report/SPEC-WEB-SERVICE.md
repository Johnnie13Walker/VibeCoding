# Командный центр продаж (веб-приложение) — спецификация

> Полноценное веб-приложение со своим интерфейсом для отдела продаж Belberry.
> Хранит историю данных ежедневных сводок, даёт календарь (дата → полный отчёт за день),
> и дашборды: по отделу, по сотрудникам, по выполнению планов. Без Looker — всё внутри приложения.
>

## Функциональность

### 1. Ежедневные сводки (ядро)
- Календарь (помесячно): доступные дни кликабельны → полный отчёт за день в новой вкладке.
- Отчёт дня = текущий формат HTML-сводки (шапка → 30 сек → Тигр → кого пинать → зависшие → активность → встречи + глубокий разбор по транскрипту → брифы/КП → паттерны → итог).
- Архив всех прошлых дней.

### 2. Дашборды (наращиваем после ядра)
- **Отдел:** воронка, суммы по стадиям, новые/выигранные/отвалы, динамика по дням/неделям.
- **Сотрудники:** активность (звонки/встречи/брифы/КП), конверсии, история по менеджеру, нагрузка.
- **План-факт:** выполнение планов (общий + per-МОП + ТМ: встречи/наборы/звонки 120с+) против факта, прогноз.

### 3. Автогенерация + доставка
- daily-runner 09:00 Пн–Пт → запись в БД + рендер отчёта + ссылка в чат менеджеров.

## Авторизация (Bitrix24-identity, passwordless)

1. Пользователь вводит свой **email** на странице входа.
2. Бэкенд проверяет email в Битрикс24 (`user.get` по EMAIL) → находит активного пользователя.
3. Генерируется одноразовый код/магик-линк, отправляется этому пользователю **сообщением от Ларисы** в Битрикс24 (`im.message.add` / `im.notify.personal.add` от пользователя 2812).
4. Пользователь вводит код / переходит по ссылке → сессия (JWT/cookie).
5. Доступ — только активным сотрудникам Битрикс24 (опц. ограничить отделом продаж + руководством). Роли (директор/РОП/менеджер) — для разграничения видимости дашбордов.

➕ Нет отдельных паролей, единая идентичность с Битрикс24, канал доставки кода уже есть (Лариса).

## Архитектура

```
┌─ daily_runner (cron Пн-Пт 09:00, Python) ──────────────┐
│  Bitrix REST + voximplant + Wazzup + транскрипты        │
│  → collect / enrich / analyze(LLM) / render             │
│  → запись в Postgres (структура дня + html отчёта)      │
│  → Telegram: ссылка /day/<date>                         │
└─────────────────────────────────────────────────────────┘
        │  (общая БД)
┌─ Next.js app (VPS, nginx, поддомен, TLS) ───────────────┐
│  /login            вход по email → код от Ларисы        │
│  /                 календарь                            │
│  /day/<date>       полный отчёт дня                     │
│  /dashboard/team   дашборд отдела                       │
│  /dashboard/people дашборд по сотрудникам               │
│  /dashboard/plan   план-факт                            │
│  /api/*            данные (защищены сессией/ролями)     │
└─────────────────────────────────────────────────────────┘
```

### Модель данных (черновик)
- `users(bitrix_id, email, name, role, dept)` + `sessions/login_codes`
- `reports(date PK, status, html, summary_json, generated_at)`
- `deals_snapshot(date, deal_id, stage, opportunity, manager_id, stuck_days, …)`
- `meetings(date, meeting_id, deal_id, type, status, manager_id, analysis_json)`
- `manager_activity(date, manager_id, calls, answered, conv, meetings_set, briefs, kp)`
- `kp_briefs(date, item_id, deal_id, type, stage, manager_id, amount)`
- `plans(period, manager_id, metric, target)` — для план-факт

## Рекомендуемый процесс и фазы (MVP-first)

Это многонедельный продукт → вести как **GSD-проект**, MVP-first. Claude планирует и собирает локально, Codex деплоит на VPS и подключает прод-интеграции.

- **MVP (Фазы 0–4):** ядро генератора → daily-runner+LLM → Postgres → Next.js (login + календарь + отчёт дня) → Telegram-ссылка. **Это уже рабочий сервис «календарь → отчёт».**
- **Фаза 5:** дашборд отдела.
- **Фаза 6:** дашборд по сотрудникам.
- **Фаза 7:** план-факт + прогноз.

## Открытые блокеры

1. **chat_id** менеджерского чата + бот-отправитель (Лев Петрович?).
2. **Хостинг:** поддомен + TLS на VPS «КлаудБот» — поднимаем под Next.js (Node) + Postgres.
3. **LLM-ключ/бюджет** стека Лев Петровича — переиспуем?
4. **Источник планов** для план-факт (общий + per-МОП + ТМ — где лежат эталонные цифры; см. план мая 2026).
5. **Роли и видимость:** кто что видит (менеджер — только себя? РОП/директор — всё?).
6. **Ретеншн** истории данных.

## Принципы

- MVP-first: рабочий «календарь → отчёт» прежде дашбордов.
- Авторизация с первого релиза (чувствительные данные).
- Не плодить сущности сверх нужного отделу; дашборды добавляем по одному.
