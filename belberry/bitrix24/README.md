# Bitrix24 Belberry

## Назначение

Этот контур нужен для наведения порядка в Bitrix24 Belberry по сделкам, задачам, регламентам и автоматизациям.

Здесь хранятся только рабочие материалы, схемы, решения, runbooks и безопасные шаблоны. Реальные выгрузки из Bitrix24, персональные данные, токены и `.env` в git не добавляются.

## Setup

Перед первым запуском в новом окружении установите runtime-зависимости:

```bash
python3 -m pip install -r belberry/bitrix24/requirements.txt --user
```

`cryptography` нужна для `providers/google_sheets.py`: модуль подписывает service account JWT через RS256 для read-only доступа к Google Sheets API. Остальная duplicate-merge логика по-прежнему опирается на stdlib.

## Структура

```text
belberry/bitrix24/
  audits/       # аудиты текущего состояния CRM
  deals/        # правила, воронки, стадии, качество сделок
  tasks/        # задачи, постановка, контроль, регулярные процессы
  automation/   # роботы, триггеры, сценарии автоматизации
  config/       # шаблоны доступов, реальные секреты не коммитятся
  docs/         # общая документация и решения
  exports/      # локальные выгрузки, не коммитятся
```

## Правила работы

- Все даты, дедлайны, расписания и отчёты вести по МСК: `Europe/Moscow`.
- Не хранить в git токены, webhook URL, `.env`, CSV/XLSX с персональными данными и сырые CRM-выгрузки.
- Доступы к Google и Bitrix24 описывать через `config/.env.example`; реальные значения хранить только в gitignored `.env` и JSON-файлах вне git.
- Перед изменением процессов фиксировать текущее состояние и ожидаемый эффект.
- Любое изменение в Bitrix24 должно быть обратимым или иметь понятный rollback-план.
- Если используется выгрузка из Bitrix24, класть её в `exports/`; эта папка исключена из git.

## Основные направления

1. Сделки:
   - аудит воронок и стадий
   - правила обязательных полей
   - контроль дублей и потерянных сделок
   - отчёты по качеству ведения CRM

2. Задачи:
   - правила постановки задач
   - контроль просрочек
   - регулярные задачи и шаблоны
   - связь задач со сделками

3. Автоматизации:
   - роботы и триггеры
   - уведомления
   - контроль SLA
   - безопасное тестирование перед включением

## Следующий шаг

Сначала нужен аудит текущего состояния:

1. Какие воронки и стадии используются.
2. Какие поля обязательны по типам сделок.
3. Какие задачи создаются вручную и автоматически.
4. Какие роботы и триггеры уже включены.
5. Где сейчас основные проблемы: дубли, просрочки, пустые поля, потерянные сделки, неработающие уведомления.
6. Какие Google Sheets/Drive-файлы должны быть доступны service account.
