## Задача 3 — боевой полный dry-run rebind-сирот (на фиксе #135)

**Контекст.** Модуль `rebind-orphan-contacts` (#129) привязывает контакты-сироты (импорт
«Вернувшийся клиент», ~3775 контактов с `COMPANY_ID=0`, `SOURCE_ID=5`) к компаниям по телефону.
**Был прод-баг** (фикс **PR #135**, влит): искал по голым 10 цифрам, Bitrix `findbycomm` их не
матчит → находил 0. Claude прогнал read-only выборки: после фикса rebindable ~10% (40 сирот → 4
матча) ⇒ ориентир **~350–400 привязываемых** из ~3775. Полный прогон Claude не делал — это тысячи
API-вызовов (~часы), лучше на сервере.

**Что сделать (на сервере, репо с влитым #135):**

1. **Подтянуть main** (должен содержать #135 `_bitrix_phone_queries`).
2. **Полный dry-run** (по умолчанию dry-run; пишет план во вкладку Sheets `contact_rebind_plan`,
   в Bitrix НЕ пишет):
   ```
   python -m crm_company_enrich.cli rebind-orphan-contacts
   ```
   Стейджить можно через `--limit N` (сначала 500 → потом весь набор). Долго (rate-limited).
3. **Разобрать план** в `contact_rebind_plan`: сколько `MATCH_COMPANY` / `MATCH_VIA_CONTACT` /
   `AMBIGUOUS` / `NO_MATCH` / `NO_PHONE`. `AMBIGUOUS` (несколько компаний на телефон) — на ручной
   разбор, не привязывать вслепую.
4. **Боевая привязка** — `--live` (пишет `COMPANY_ID` контакта + привязывает к открытым сделкам
   компании). Только bind/update, **не удаление** → killswitch `BITRIX_ALLOW_DELETE` НЕ требуется.
   Рекомендуется сначала `--live --limit 50` (smoke), проверить в Bitrix, потом весь набор.

**Осторожно:** `--live` пишет в прод-Bitrix. Сверять план до запуска; `AMBIGUOUS` исключать.
