## 1. Главная цель проекта

Строится персональный AI-ассистент Cloudbot на базе OpenClaw/OpenClo с интеграциями в Telegram, Bitrix24, задачи, календарь, отчёты, поиск, DevOps и бизнес-автоматизации.

Бизнес-ценность:
- снизить ручной операционный контроль пользователя;
- автоматизировать CRM-процессы, отчёты, проверки и рутинные действия;
- обеспечить стабильный ежедневный контроль системы и бизнес-процессов;
- ускорить работу с Bitrix24, особенно по качеству данных, дублям, обогащению и CRM-гигиене.

Текущий активный бизнес-фокус: безопасное объединение дублей сделок Bitrix24 в воронке `Реанимация` с сохранением коммуникаций, документов, комментариев, контактов, источников и UTM.

3. Создать backup перед merge.
4. Нормализовать только разрешённые поля для снятия типовых Bitrix merge-конфликтов.
5. Выполнить штатный `crm.entity.mergeBatch`.
6. Проверить результат.
7. Успешные группы перенести в лист `Объединено`.
8. Конфликты оставить в рабочем листе со ссылкой `Открыть merge`.

## 4. Важные технические решения

Принято:
- Использовать штатное слияние Bitrix24, не удалять сделки вручную.
- Перед каждым merge делать JSON backup.
- Основная сделка выбирается по политике:
  - успех/активная продажа важнее;
  - аккаунтинг не трогать;
  - в телемаркетинге без пересечений брать свежую сделку;
  - источник и UTM брать с самой ранней сделки.
- Не пытаться объединять группы с разными компаниями автоматически.
- Не пытаться объединять успехи по разным продуктам.
- При конфликте Bitrix не форсировать merge, а оставлять ссылку на ручное объединение.
- Google Sheet является операционной очередью: рабочий лист `Дубли Реанимация`, архив `Объединено`.

Решено НЕ делать:
- Не удалять сделки вместо merge.
- Не переписывать всю архитектуру CRM-автоматизации ради одной задачи.
- Не объединять рискованные группы без ручной проверки.
- Не менять аккаунтинговые сделки.

Компромисс:
- Автоматизация снимает типовые конфликты только по allow-list полей, но не пытается решать все бизнес-конфликты.
- Часть дублей остаётся для ручного merge, зато снижается риск потери данных.

## 7. Известные проблемы и подводные камни

- Нельзя удалять сделки: только merge.
- Нельзя автоматически трогать аккаунтинговые сделки.
- Нельзя объединять успехи по разным продуктам.
- В Bitrix merge может не пройти даже после нормализации allow-list полей.
- Группы с разными компаниями должны уходить в `Ручная проверка`.
- Источник/UTM должны сохраняться с самой ранней сделки, а не с основной.
- Основная сделка в таблице должна быть ссылкой, не просто ID.
- После успешного merge исходные сделки могут стать `Not found`; это ожидаемо.
- Google Sheet timezone в metadata: `America/Los_Angeles`, но все рабочие отметки ставятся в МСК вручную.
- Локальный Google service account не имеет доступа к текущей таблице.
- После прерываний нужно проверять фоновые процессы. На момент handoff фоновых Bitrix merge-процессов не найдено.

## 9. Бизнес-контекст

В Bitrix есть несколько воронок/контекстов:
- Реанимация.
- Продажи / ОП.
- Телемаркетинг.
- Аккаунтинг.
- Успех.

CRM-логика дублей:
- Дубль: 2+ сделок с одинаковым нормализованным доменом/названием сайта.
- Нормализация: регистр, `http/https`, `www`, конечные слэши.
- Если есть сделка в успехе, она обычно основная.
- Несколько успехов по разным продуктам не объединять.
- Аккаунтинговые сделки не трогать.
- Если есть активная продажа или `отложено`, она основная.
- Если только ТМ/реанимация без продаж/успеха/аккаунтинга, объединять в свежую сделку.
- Источник и UTM брать с самой ранней сделки.
- Коммуникации, документы, комментарии и активности должны сохраняться через штатный Bitrix merge.
