# Шаблон архитектурной ревизии

Использовать для ревизии отдельного агента, workflow, runtime-контура или всего репозитория.

## 1. Инвентаризация контуров

Заполнить:

- Что является предметом ревизии.
- Какие пользовательские сценарии затронуты.
- Какие технические контуры задействованы: Telegram, cron, manual run, deploy, bridge, health-check.
- Какие файлы и директории являются центральными для этого контура.

## 2. Канонический репозиторий

Заполнить:

- Локальный путь канонического runtime-репозитория.
- Git remote, который считается source of truth.
- Рабочая ветка или release-ветка.
- Есть ли расхождение между локальным репозиторием, GitHub и сервером.

## 3. Канонический runtime

Заполнить:

- Где именно исполняется контур.
- Через какой entrypoint он запускается в production.
- Какие env/config/state используются.
- Есть ли bridge, shell wrapper, cron или другой промежуточный слой.

## 4. Реальные точки входа

Заполнить по каждому пути:

- Telegram path.
- Cron path.
- Manual run path.
- Deploy path.
- Recovery / rollback path.

Для каждого пути указать:

- вход;
- маршрутизацию;
- исполняемый workflow;
- агент или provider;
- выходной артефакт.

## 5. Карта зависимостей

Заполнить:

- Какие workflows зависят от каких агентов.
- Какие агенты зависят от каких providers.
- Какие providers зависят от env и внешних API.
- Какие shell-скрипты или cron-записи запускают этот контур.

## 6. Архитектурные проблемы

Заполнить:

- Где есть дублирование.
- Где нарушены границы слоев.
- Где legacy смешан с live-контуром.
- Где source of truth неочевиден.
- Где есть риск расхождения GitHub, локальной машины и сервера.

## 7. Корневые причины

Заполнить:

- Какая проблема является симптомом.
- Какая причина реально порождает проблему.
- Почему проблема не решается локальным workaround.
- Что именно должно измениться на архитектурном уровне.

## 8. Целевая схема

Заполнить:

- Какой путь должен стать каноническим.
- Какие контуры должны быть выведены в archive или migration.
- Где должны жить config, env, state и runtime wiring.
- Какой слой должен владеть каждой частью сценария.

## 9. План исправления

Заполнить:

- Что исправить сначала.
- Что нельзя менять в этой итерации.
- Какие файлы и контуры затронет исправление.
- Какой rollback возможен.
- Какой техдолг останется после итерации.

## 10. Как проверять

Заполнить:

- Какие тесты запустить.
- Какой smoke-check обязателен.
- Какой ручной сценарий нужно проверить.
- Какие логи, отчеты или метрики нужно посмотреть.
- По какому критерию Архитектор принимает результат.
