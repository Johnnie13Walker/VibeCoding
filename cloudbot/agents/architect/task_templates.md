# Шаблоны задач для инженера

## 1. Архитектурная ревизия

### Когда использовать

Когда нужно понять, как реально устроен контур, где дублирование, где legacy и где нарушены границы слоев.

### Формулировка задачи

Проведи архитектурную ревизию контура и собери фактическую карту исполнения.

Проверь:

- `cloudbot/orchestrator/*`;
- `cloudbot/workflows/*`;
- `agents/*`;
- `cloudbot/providers/*`;
- `infra/orchestrator/*`;
- `configs/schedules.cron`;
- связанные `scripts/*` и runtime-артефакты.

На выходе дай:

- инвентаризацию entrypoints;
- канонический путь исполнения;
- найденные дубли и legacy-пути;
- корневую причину архитектурных проблем;
- целевую схему исправления.

### Критерии готовности

- Назван канонический runtime.
- Назван source of truth.
- Названы реальные точки входа.
- Корневая причина сформулирована явно.
- Есть конкретный план исправления.

### Обязательная регрессия

- Проверка затронутого smoke-сценария.
- Проверка логов или отчетов.
- Подтверждение, что критичный сценарий не ушел в альтернативный путь.

## 2. Починка сломанного контура

### Когда использовать

Когда сценарий перестал работать в Telegram, cron, bridge, deploy или server runtime.

### Формулировка задачи

Локализуй точку разрыва контура и восстанови канонический путь исполнения.

Проверь последовательно:

- входной trigger;
- router или shell wrapper;
- workflow;
- agent или bridge;
- provider;
- env/config/state;
- лог ошибки.

Не ограничивайся локальным фиксированием симптома. Укажи, почему контур сломался именно здесь.

### Критерии готовности

- Названа точка разрыва.
- Исправление внесено в канонический путь, а не в обходной.
- Подтверждено, что альтернативный broken path не остался активным.
- Зафиксирован риск повторения.

### Обязательная регрессия

- Тесты по затронутому контуру.
- Ручной прогон критичного сценария.
- Проверка логов после исправления.

## 3. Контроль изменений перед merge или deploy

### Когда использовать

Перед merge в целевую ветку, перед deploy на сервер или перед включением cron/schedule.

### Формулировка задачи

Проведи change-control для изменения и определи, можно ли его безопасно принимать.

Нужно зафиксировать:

- какие файлы и контуры затронуты;
- какой runtime-путь меняется;
- какие инварианты нельзя сломать;
- какой smoke-check обязателен;
- какие env/config/state завязаны на изменение;
- какой rollback или fallback предусмотрен.

### Критерии готовности

- Есть список затронутых контуров.
- Есть архитектурные риски.
- Есть обязательные проверки до и после deploy.
- Есть решение: `approve`, `approve with conditions` или `reject`.

### Обязательная регрессия

- Dry-run, если он предусмотрен.
- Smoke-check боевого сценария.
- Проверка логов и отчетов после применения изменения.

## 4. Ревизия нового агента

### Когда использовать

Когда в систему добавляют новый агент или расширяют старый агент до отдельной сущности.

### Формулировка задачи

Проверь, что новый агент встроен в систему как отдельный контур, а не как набор случайных файлов.

Нужно проверить:

- есть ли у агента явная зона ответственности;
- есть ли понятный entrypoint;
- есть ли границы между agent, workflow и provider;
- нет ли прямого обхода через Telegram, env или чужой runtime;
- есть ли критерии приемки и проверок.

### Критерии готовности

- У агента есть собственный пакет документов или кодовый модуль.
- Его role не дублирует существующий агент без причины.
- Канонический путь вызова определен.
- Границы ответственности задокументированы.

### Обязательная регрессия

- Проверка вызова через канонический workflow.
- Проверка, что соседние агенты не сломаны.
- Проверка логов/отчетов, если агент участвует в runtime.

## 5. Ревизия runtime-контура

### Когда использовать

Когда нужно подтвердить, что cron, Telegram, bridge, deploy и ручной запуск согласованы между собой.

### Формулировка задачи

Собери карту runtime-контура и проверь согласованность всех путей исполнения.

Нужно сверить:

- `cloudbot/bot/telegram/*`;
- `cloudbot/orchestrator/*`;
- `infra/orchestrator/run_workflow.sh`;
- `infra/orchestrator/workflows/*.sh`;
- `configs/schedules.cron`;
- `Makefile`;
- bridge-скрипты в `scripts/*`;
- server env/state, если контур зависит от сервера.

### Критерии готовности

- Для Telegram, cron, manual run и deploy названы entrypoints.
- Названы расхождения между путями.
- Понятно, какой путь считается каноническим.
- Есть список обязательных исправлений, если пути расходятся.

### Обязательная регрессия

- Проверка не менее одного сценария из Telegram.
- Проверка не менее одного cron-сценария.
- Проверка deploy или post-change verify, если путь затронут.
