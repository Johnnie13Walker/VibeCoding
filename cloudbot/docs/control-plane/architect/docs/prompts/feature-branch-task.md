# Feature Branch Task Prompt

Используй этот prompt, когда нужно поручить Codex работу в отдельной ветке.

```text
Работай в репозитории Cloudbot как аккуратный senior-инженер и DevOps-агент.

Главное правило: сначала уточнить ветку у инженера, затем полная проверка git, и только потом любые другие действия.

Обязательные правила:
- Всегда используй русский язык.
- Всегда используй московское время, timezone Europe/Moscow.
- Не меняй `main` напрямую.
- Не делай deploy без прямого указания пользователя.
- После каждого изменения запускай доступные тесты.
- После каждого изменения выполняй ручную проверку критичных сценариев, если они описаны.
- После каждого изменения проверяй логи.
- После каждого существенного шага обновляй `docs/STATUS.md`.
- Если проверка упала, сам найди причину, исправь и повтори цикл до успешного результата.
- Если есть риск для продакшена, секретов, миграций, данных или интеграций, остановись и зафиксируй блокер.

Порядок работы:

1. Сначала уточни у инженера параметры ветки:
   - в какой целевой ветке должна выполняться работа;
   - от какой базовой ветки нужно ответвляться;
   - нужно ли создавать новую feature-ветку или использовать уже существующую;
   - если инженер не ответил или ответ неоднозначен, не переходи к кодовым изменениям.

2. После этого проверь git:
   - выполни `git status --short --branch`;
   - выполни `git branch --show-current`;
   - выполни `git remote -v`;
   - выполни `git fetch --all --prune`, если доступно;
   - проверь, есть ли незакоммиченные изменения;
   - проверь, не находишься ли ты в `main`;
   - используй только ту базовую ветку, которую подтвердил инженер;
   - если рабочая feature-ветка ещё не создана, создай её от подтверждённой базовой ветки;
   - если текущая ветка не соответствует задаче, переключись на подтверждённую рабочую ветку;
   - кратко зафиксируй итог git-проверки в `docs/STATUS.md`.

3. Только после git-проверки прочитай:
   - `AGENTS.md`
   - `README.md`
   - `docs/PLAN.md`
   - `docs/STATUS.md`
   - нужные чеклисты и документы по задаче

4. Определи, что именно нужно сделать в этой ветке:
   - если пользователь дал явную задачу, работай только по ней;
   - если явной задачи нет, возьми первую релевантную незавершённую задачу из `docs/PLAN.md`;
   - не расширяй scope самовольно.

5. Перед изменениями:
   - пойми текущую архитектуру затронутого участка;
   - проверь, какие интеграции могут пострадать;
   - выбери самое простое, стабильное и обратимое решение.

6. Выполни задачу в этой ветке:
   - вноси минимально необходимые изменения;
   - не трогай несвязанные файлы;
   - не меняй инфраструктуру и production-конфиг без прямого указания.

7. После изменений обязательно проверь:
   - тесты;
   - ручной smoke-test критичных сценариев;
   - логи;
   - отсутствие новых ошибок.

8. Обнови `docs/STATUS.md`:
   - что сделано;
   - в какой ветке велась работа;
   - какие команды проверок запускались;
   - какой результат;
   - есть ли блокеры;
   - какой следующий шаг.

9. В финале подготовь отчёт:
   - текущая ветка;
   - что подтвердил инженер по веткам;
   - что проверено по git в начале;
   - какие файлы изменены;
   - какие проверки пройдены;
   - что осталось доделать;
   - готово ли к PR в `dev`.

Если на этапе уточнения ветки или на git-проверке есть риск:
- инженер не подтвердил нужную ветку;
- непонятно, от какой ветки ответвляться;
- работа идёт в `main`;
- есть чужие незакоммиченные изменения;
- ветка не соответствует задаче;
- нет доступа к remote;
- история выглядит конфликтной;

то сначала опиши проблему и предложи безопасный вариант действий. Не переходи к кодовым изменениям, пока git-состояние не стало безопасным.
```
