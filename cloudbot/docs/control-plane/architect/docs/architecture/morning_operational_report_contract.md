# Контракт единого утреннего operational-отчёта

Документ фиксирует канонический формат единого утреннего отчёта Ларисы Ивановны по техническому состоянию Cloudbot/OpenClaw/OpenCloud-контура.

Цель:
- убрать дубли между `healthcheck-daily`, кратким статусом и infra-summary;
- оставить пользователю один понятный утренний отчёт;
- не путать состояние API, состояние платформы и фактическую доступность пользовательских сценариев.

Все времена в отчёте указываются в `Europe/Moscow`.

## Принцип одного утреннего отчёта

Пользователю должен приходить один итоговый operational-отчёт в `09:30 МСК`.

Внутренние upstream-проверки могут выполняться раньше, но не должны превращаться в отдельные пользовательские сообщения при зелёном статусе:
- `healthcheck-daily` остаётся внутренним диагностическим источником;
- промежуточный краткий статус не отправляется отдельным сообщением;
- infra-summary не отправляется отдельным сообщением после already-final статуса.

Исключение:
- отдельное сообщение допустимо только при активной аварии, если нужна немедленная ранняя эскалация до финального отчёта.

## Каноническая шапка

В ежедневном отчёте допустимы только два верхнеуровневых статуса:

- `🟢 СЕРВЕР СТАБИЛЕН`
- `🔴 ЕСТЬ ПРОБЛЕМЫ`

Формулировки вида `ОК`, `всё работает`, `critical issues not found` допустимы только внутри блока `🧠 Итог`, но не как отдельный самостоятельный отчёт.

## Каноническая структура отчёта

Следующие блоки показываются всегда:

1. `🧠 Итог`
2. `🖥 Хост`
3. `☁️ OpenClaw / OpenCloud`
4. `🔌 API и интеграции`
5. `🧩 Пользовательские возможности`
6. `⏱ Планировщик и доставка`
7. `🛡 Безопасность`
8. `📌 Наблюдения / техдолг`
9. `📊 Индекс здоровья`
10. `🕒 Время отчёта`

Следующие блоки показываются условно:

- `🆕 Новые API` — только если реально появились новые интеграции, ещё не переведённые в штатный контур;
- `🚨 Активные проблемы` — только если есть текущие/неустранённые сбои.

## Что считается API, а что capability

Отчёт обязан разделять три разные плоскости:

### 1. Платформенный контур

Это состояние OpenClaw/OpenCloud как платформы:
- gateway reachable;
- gateway probe;
- deep diagnostics;
- cron API;
- delivery path.

### 2. API и интеграции

Это доступность внешних и внутренних интеграций:
- Telegram;
- OpenAI;
- Bitrix portal;
- Bitrix OAuth;
- Todoist;
- WHOOP;
- WAZZUP;
- WEBHOOK;
- Web Search provider;
- другие зафиксированные интеграции.

API-статус отвечает только на вопрос:
- доступна ли интеграция как технический ресурс.

### 3. Пользовательские возможности

Это фактическая доступность сценариев для пользователя:
- дневной бриф;
- задачи;
- встречи;
- погода;
- web search для Ларисы;
- другие реальные user-facing сценарии.

Capability-статус отвечает на вопрос:
- может ли Лариса реально выполнить пользовательский сценарий end-to-end.

Нельзя подменять capability-статус API-статусом.

Пример неправильной интерпретации:
- `Web Search provider = OK` не означает `Web search для Ларисы = OK`.

Если search provider жив, но у Ларисы нет user-facing route/skill-path, то:
- в `🔌 API и интеграции` можно показать `Web Search provider: OK`;
- в `🧩 Пользовательские возможности` обязательно должно быть `Web search для Ларисы: НЕ НАСТРОЕН` или `degraded`.

## Правила для блока "Новые API"

Блок `🆕 Новые API` должен использоваться строго как переходный.

Интеграция может находиться в состоянии:
- `new` — впервые обнаружена, ещё не классифицирована;
- `known` — подтверждена и переведена в штатный контур;
- `deprecated` — выведена из эксплуатации;
- `degraded` — известна, но сейчас работает нестабильно;
- `failed` — известна, но сейчас недоступна.

Правила:

1. Если интеграция впервые обнаружена и ещё не занесена в реестр штатных интеграций, она попадает в `🆕 Новые API`.
2. После подтверждения она должна перейти в основной блок `🔌 API и интеграции`.
3. Одна и та же интеграция не должна ежедневно повторяться в `🆕 Новые API`.
4. Если интеграция давно штатная, но сейчас деградировала, она остаётся в основном блоке API со статусом `degraded` или `fail`, а не возвращается в `🆕 Новые API`.

Применение к текущему контуру:
- `WAZZUP`
- `WAZZUP_WEBHOOK_FORWARD`
- `WEBHOOK`

если они обнаружены и подтверждены давно, должны всегда жить в основном блоке `🔌 API и интеграции`, а не в `🆕 Новые API`.

## Канонические статусы

Для API, платформы и capability используются только статусы:
- `OK`
- `degraded`
- `fail`
- `not configured`
- `not checked`

Правила:
- `OK` — реальная проверка успешна;
- `degraded` — функция доступна частично, с ограничением или нестабильностью;
- `fail` — текущий сценарий не работает;
- `not configured` — сценарий или интеграция не настроены;
- `not checked` — интеграция известна, но в этом прогоне не была проверена.

Нельзя:
- молча пропускать известную интеграцию из отчёта;
- писать `OK`, если была проверена только конфигурация, но не реальная функция;
- выдавать stub/заглушку за рабочий end-to-end сценарий.

## Web Search: как должно работать правильно

Web search обязан иметь три независимых статуса.

### A. Search provider

Проверяет:
- что provider выбран;
- что конфигурация валидна;
- что тестовый запрос реально может выполниться;
- что вернулся хотя бы один короткий результат с источником.

Это статус блока `🔌 API и интеграции`.

### B. Search skill

Проверяет:
- что skill `web_search` не является заглушкой;
- что skill вызывает реальный provider;
- что skill возвращает результат, а не только `ok: true`;
- что в ответе есть provider/source/evidence.

### C. Web search для Ларисы

Проверяет:
- что у Ларисы есть user-facing route/intent/command;
- что запрос пользователя проходит через нужный workflow;
- что результат доходит до ответа Ларисы;
- что при отсутствии настройки Лариса даёт корректное диагностическое сообщение, а не ложный `OK`.

Это статус блока `🧩 Пользовательские возможности`.

Правильная интерпретация:
- provider `OK` + skill `OK` + user-facing route отсутствует = capability `not configured`;
- provider `OK`, но тестовый search smoke падает = API `degraded` или `fail`;
- user-facing route есть, но skill возвращает stub = capability не `OK`, а `degraded` или `fail`.

## Что считается активной проблемой

Активной проблемой считается только текущий или неустранённый сбой.

Не считаются активной поломкой без дополнительных признаков:
- `missing scope: operator.read`, если базовый gateway/cron path жив;
- `multi_user_heuristic`, если это архитектурный warning без фактических отказов;
- доступное обновление OpenClaw без текущей деградации;
- единичный self-healed error у cron/job;
- отсутствие `systemd user service`, если боевой контур при этом реально работает.

Такие случаи должны попадать в `📌 Наблюдения / техдолг`.

## Что должно остаться внутренним техлогом

Не нужно ежедневно слать пользователю отдельным сообщением:
- полный read-only `healthcheck-daily`;
- промежуточное `ОК. За последние 24 часа...`;
- отдельный infra-summary после уже отправленного финального отчёта.

Внутренне можно сохранять:
- raw payload health-check;
- детали deep-probe;
- расшифровку retry/self-healed cron incidents;
- служебные артефакты smoke-проверок.

Пользователю отправляется только финальный рендер.
