# Bitrix24: директорское заключение по воронкам

Дата данных: `2026-04-09 08:58 МСК`  
Основание: live-аудит `Bitrix24` по сделкам, лидам и смарт-процессам.

## 1. Итоговый вердикт

Bitrix24 в компании не пустой и не формальный. Система реально используется, в ней накоплен большой рабочий массив, и команды на неё опираются.

Но с управленческой точки зрения текущая архитектура воронок устроена неправильно.

Главная проблема: в одном CRM-контуре смешаны разные функции бизнеса:

- новые продажи;
- телемаркетинг;
- реактивация базы;
- аккаунтинг и retention;
- производство и проекты;
- архивы;
- технические процессы и датасеты;
- внутренние HR-процессы.

Из-за этого Bitrix24 сейчас помогает работать, но плохо помогает управлять.

## 2. Что руководству важно знать

### 1. Основная коммерческая воронка есть, но вокруг неё вырос хаос

Воронка `Продажи` выглядит как рабочий new business pipeline и может быть сохранена как ядро коммерческого контура.

Одновременно рядом существуют отдельные и частично пересекающиеся воронки:

- `Телемаркетинг`
- `Реанимация`
- `Аккаунтинг OLD`
- `Аккаунтинг NEW`
- `Retention - Отвалы`
- `Retention - Оплачено`
- `Проекты`

Это уже не единая архитектура, а исторически сложившийся набор контуров.

### 2. Самая проблемная зона — `Реанимация`

Это крупнейший риск для управляемости:

- `6237` карточек;
- `6124` открытых;
- `19` стадий;
- внутри смешаны продажи, лиды, база бывших клиентов, чёрные списки и причины отказов.

По сути это не воронка, а большой склад сегментов и сценариев.

### 3. Аккаунтинг и retention раздроблены

Сейчас постпродажный контур размазан между несколькими категориями:

- `Аккаунтинг OLD`
- `Аккаунтинг NEW`
- `Retention - Отвалы`
- `Retention - Оплачено`

Это означает отсутствие одного стандарта работы с текущими клиентами, upsell, churn-risk и возвратом.

### 4. Production частично живёт внутри продаж

Категория `Проекты` находится в сделках, хотя по смыслу это уже delivery-контур, а не new business.

Это искажает коммерческую аналитику и смешивает продажу с исполнением.

### 5. Архивные и технические сущности мешают управлению

В системе присутствуют контуры, которые должны быть либо скрыты, либо строго техническими:

- `АРХИВ Общая`
- `АРХИВ ТП - Тикеты`
- `Архив_Дизайн`
- `Данные для дашборда`
- `Данные для дашборда (стадии сделки)`

Пока они находятся рядом с рабочими процессами, у руководства и у команды нет чистой картины.

## 3. Что в системе хорошо

- Источник входящего потока в верхней части воронки фиксируется качественно.
- Основная воронка `Продажи` уже существует и не требует сноса.
- Смарт-процессы `Встречи` и `Брифы` реально полезны как связка sales и производства.
- HR и часть production уже вынесены в отдельные смарт-процессы, то есть основа для нормальной архитектуры есть.

## 4. Что в системе плохо

| Проблема | Почему это критично |
| --- | --- |
| Смешение разных бизнес-функций в сделках | Нельзя прозрачно управлять ни продажами, ни delivery, ни customer success |
| Дубли контуров | Возникают два стандарта работы и два набора KPI |
| Архивы остаются рабочими | Руководство видит шум вместо реальной операционной картины |
| Слишком длинные воронки | Стадии перестают быть управленческими решениями и превращаются в склад |
| Технические сущности рядом с бизнес-воронками | Аналитика компенсирует плохую архитектуру вместо того, чтобы опираться на неё |

## 5. Правильная целевая архитектура

Рекомендуемая модель:

| Контур | Решение |
| --- | --- |
| New business | Оставить одну основную воронку `Продажи` |
| SDR / первичный обзвон | Оставить отдельный короткий контур `Телемаркетинг` с обязательной передачей в `Продажи` |
| Реактивация | Пересобрать `Реанимацию` в короткую воронку на `4-6` стадий, сегменты вынести в поля |
| Аккаунтинг / upsell / retention | Объединить текущие OLD/NEW/Retention-контуры в единую post-sale архитектуру |
| Delivery | Вынести `Проекты` из сделок в отдельный smart-process `Проект / Delivery` |
| Production | Оставить `Брифы`, `Копирайтинг`, `Биржа`, но привести их к роли именно производственных процессов |
| HR | Оставить отдельным внутренним контуром вне коммерческой CRM-архитектуры |
| Технические датасеты | Оставить только как technical layer, убрать из бизнес-навигации |
| Архив | Полностью очистить и скрыть из активного управления |

## 6. Рекомендация руководству

Правильное решение — не «подкручивать» текущую схему, а провести мягкую архитектурную нормализацию:

1. Зафиксировать владельцев контуров.
2. Запретить создание новых воронок без управленческой функции и owner.
3. Сократить коммерческий контур до понятной карты:
   - вход;
   - продажи;
   - реактивация;
   - постпродажное развитие;
   - delivery.
4. Архивы и технические сущности убрать из повседневного поля зрения команды.
5. Через обновлённую архитектуру перестроить отчётность и управленческие дашборды.

## 7. Что нужно сделать в первую очередь

| Приоритет | Действие |
| --- | --- |
| Высокий | Утвердить целевую карту воронок и owners |
| Высокий | Объединить аккаунтинговые контуры |
| Высокий | Пересобрать `Реанимацию` |
| Высокий | Вынести `Проекты` из `Сделок` |
| Средний | Закрыть и скрыть архивные категории |
| Средний | Убрать технические сущности из бизнес-меню |
| Средний | Пересобрать `Телемаркетинг` под реальную SDR-логику |

## 8. Финальное управленческое заключение

Bitrix24 можно привести к сильной и прозрачной модели без радикального сноса.

Сносить систему не нужно. Нужно убрать смешение функций.

Если оставить всё как есть, компания будет продолжать работать в Bitrix24, но руководству будет всё сложнее:

- понимать реальный pipeline;
- управлять конверсией;
- видеть ответственность команд;
- отделять продажи от производства;
- строить честную аналитику.

Если провести предложенную нормализацию, Bitrix24 станет:

- прозрачным для коммерческого директора;
- управляемым для операционного директора;
- аналитически пригодным для маркетинга;
- и гораздо более устойчивым как бизнес-система.
