# Контракт deploy/release для Cloudbot / OpenClo / Codex

## 1. Назначение документа

Этот документ фиксирует текущую и целевую модель deploy/release для Cloudbot / OpenClo / Codex, чтобы было понятно:

- откуда реально берется код для live-контуров;
- какая ветка или ref считаются источником интеграции и релиза;
- как код попадает на сервер;
- где проходит граница между `code`, `config`, `state`, `artifacts`;
- почему текущая схема ведет себя неуправляемо;
- какая минимальная схема нужна, чтобы вернуть управляемость без выдуманного CI/CD.

Документ опирается на два типа источников:

- `Подтверждено локально`:
  - текущий репозиторий `/Users/pro2kuror/Desktop/architect`;
  - инженерный git-репозиторий `/Users/pro2kuror/Desktop/OpenClo/projects/engineer`;
  - в `projects/engineer` существует `.git`, `origin=https://github.com/Johnnie13Walker/codex-base.git`, текущий `HEAD=c04443f8eb46985e33c1127a09eb804bcbfd5eb7`;
  - в `projects/engineer` есть локальные и удаленные ветки `main`, `dev` и множество `feature/*`, `codex/*`;

- фактический deploy-скрипт кроме шаблона `scripts/deploy.sh`;
- способ, которым сервер сейчас хранит версию live-кода.

## 2. Охват: какие контуры покрывает документ

Документ покрывает только те deploy/release-контуры, которые влияют на live-runtime:

1. Cloudbot / agent-код, который развивается локально и публикуется в GitHub.
2. Server OpenClaw core, живущий отдельно от локального cloudbot-репозитория.
3. Cron / scheduled сценарии и их кодовые источники.
4. Tar-synced агентские каталоги и их снапшоты.
5. Server `config/env/state/artifacts`.
6. Документированный контур управления текущего репозитория.

Документ не утверждает наличие CI/CD, реестра артефактов или release pipeline там, где они не подтверждены.

### 3.3 Практический вывод по текущей реальности

Следствие: формально кажется, что deploy управляется через `dev -> main -> scripts/deploy.sh`, но по факту это неверная картина.

1. GitHub/локальный git-контур `projects/engineer`.

3. Tar-sync снапшоты агентских каталогов.

Это и есть ключевая причина неуправляемости release path.

## 5. Разделение `code` / `config` / `state` / `artifacts`

| Тип | Что сюда относится | Должно жить в git | Не должно жить в git | Что нужно документировать как внешний контракт |
| --- | --- | --- | --- | --- |
| `code` | исходники Cloudbot/agent-кода, deploy-скрипты, provider-код, server OpenClaw core code | Да | Нет | Какие repo канонические для каждого контура |
| `config` | не-secret конфиги, branch/release policy, deploy runbook, service templates, env schema без значений | Да | Секретные значения нет | Где и как хранятся реальные server env/config |
| `state` | runtime state, очереди, БД, кэши, last-run markers, логи, generated reports | Нет | Да | Где хранится state, кто его владелец, что является source of truth |
| `artifact` | tar-snapshot, release bundle, build archive, manifest deployed version, backup archive | Частично: схема и format да, сами generated артефакты обычно нет | Да, если это mutable или бинарные артефакты live-среды | Как собирается artifact, из какого ref, где хранится, как проверяется |

### Практическая граница

- `code` должен деплоиться как версия, привязанная к repo/ref/commit.
- `config` должен быть отделен от `code`, а секреты не должны попадать в git.
- `state` не должен переезжать как часть deploy-кода.
- `artifact` не должен быть “магическим слепком с ноутбука” без версии и происхождения.

## 6. Текущая веточная и релизная модель

### 6.1 Что декларируется документами

Подтверждено локально:

- `AGENTS.md` и `docs/workflows/codex-github.md` объявляют:
  - `dev` как основную ветку разработки;
  - `main` как стабильную ветку;
  - feature-ветки как рабочий путь изменений.

### 6.2 Что фактически нарушено

Подтверждено по ревизии:

- `main` и `dev` не отражают реального состояния проекта;
- активная разработка живет в feature-ветках;
- deploy path тянул `dev` по умолчанию;
- следовательно, документированная branch-модель не является надежной моделью релиза.

### 6.3 Фактический вывод

Сейчас у системы нет надежно определенных:

- ветка интеграции;
- stable/release branch;
- ветка-источник для deploy.

Документы говорят, что это `dev` и `main`, но operational reality говорит обратное.

Поэтому “случайный `dev` по умолчанию” нельзя сохранять:

- `dev` не является подтвержденным источником интеграции;
- default pull ветки без явного ref может выкатить не тот код;
- feature-ветка может содержать более актуальное состояние, чем `dev`, что разрушает смысл `dev` как integration branch;
- сервер после deploy не дает простой ответ, какой именно commit сейчас в проде.

## 7. Основные проблемы текущей схемы

Критичные проблемы:

1. Hardcoded или implicit pull `dev` по умолчанию.
2. Deploy идет не из доказуемого канонического источника.
3. Tar-sync с ноутбука выступает live-механизмом доставки.
4. По серверу нельзя быстро и надежно определить, какой commit/ref сейчас в проде.
5. `code`, `config`, `state` и `artifacts` смешаны логически и операционно.
6. Часть live runtime живет вне GitHub-репозитория cloudbot-кода.
7. Нет явного контракта артефакта для agent snapshots.
8. Высокий риск выкатить не ту ветку или грязную локальную копию.

Что дополнительно создает ложное ощущение управляемости:

- наличие `scripts/deploy.sh` в текущем repo, хотя это только шаблон;
- наличие описанной модели `main/dev`, хотя она не совпадает с фактической жизнью проекта;
- git-термины используются как будто все live-контуры собираются из одного repo, но это не так.

## 8. Временно допустимая схема

### 8.1 Главный принцип

Временно допустимая схема должна быть не красивой, а проверяемой.

Минимально правильная переходная модель:

- для cloudbot/agent-кода есть один явный канонический repo;
- deploy всегда идет от явного `ref`, а не “по умолчанию из `dev`”;
- сервер хранит информацию о deployed version;
- `env/state` явно считаются внешней server-реальностью;
- tar-sync допускается только как переходный transport, но не как каноническая модель источника.

### 8.2 Временный канонический источник

Для cloudbot/agent-кода временно признать каноническим source:

- repo: `Johnnie13Walker/codex-base`;
- рабочий локальный контур: `projects/engineer`;
- deploy source: только явный branch/tag/commit из этого repo.

Важно:

- `main` и `dev` в их текущем состоянии временно не считаются каноническими источниками релиза;
- пока они не приведены в порядок, нельзя считать, что “deploy from dev” означает controlled release.

### 8.3 Временная веточная и релизная модель

Рекомендуемая временная модель:

1. Feature-ветки остаются зоной активной разработки.
2. Ввести одну явную ветку интеграции: `integration`.
3. Все feature-ветки попадают только в `integration`.
4. Stable/release source временно оформлять не через “случайный `main`”, а через явный release tag, созданный от `integration`.
5. Deploy на server/live допускается только от:
   - release tag;
   - или конкретного commit SHA из `integration`, если релизная дисциплина еще не доведена до тегов.

Почему так:

- это быстрее, чем сразу чинить весь `main/dev` мир;
- это сразу убирает неявный `dev` pull;
- это дает один источник интеграции;
- это позволяет однозначно записать версию на сервере.

### 8.4 Временная схема для server OpenClaw core

`/opt/openclaw` должен временно считаться отдельным deploy-контуром с отдельным контрактом:

- у него должен быть свой repo/ref/commit;
- его нельзя описывать как будто это часть того же deploy path, что и cloudbot/agent-код;
- его версия тоже должна фиксироваться отдельно.

Пока контракт не оформлен, его нужно считать внешней runtime-зависимостью, а не частью “понятного git deploy”.

### 8.5 Временная схема для tar-sync агентских каталогов

Tar-sync временно допустим только если соблюдаются все правила:

1. Источник снапшота собирается из явного git `ref`.
2. Снапшот собирается из чистого дерева без незакоммиченных изменений.
3. В каждый снапшот вкладывается `REVISION` или `release.json` с полями:
   - `repo`
   - `branch`
   - `tag`
   - `commit`
   - `built_at_msk`
   - `dirty=false`
4. На сервере сохраняется этот manifest рядом с развернутым каталогом.
5. Tar-sync не считается источником релиза, он только способ переноса уже определенного artifact.

## 9. Целевая схема

### 9.1 Целевая зрелая веточная и релизная модель

После стабилизации веточной истории:

1. `feature/*` -> PR -> `dev` как ветка интеграции.
2. Из `dev` после проверки создается release tag.
3. `main` отражает только released/stable состояние.
4. Deploy идет не из “текущей ветки на ноутбуке”, а из зафиксированного release artifact или зафиксированного git ref.

Если проекту удобнее оставить отдельную `release` branch вместо `main`, это допустимо, но роли должны быть такими же:

- одна ветка интеграции;
- одна stable/release branch;
- immutable release tags;
- deploy только из явного источника релиза.

### 9.2 Целевая модель доставки

Целевая зрелая схема:

- cloudbot/agent-код деплоится из канонического git-источника или из artifact pipeline, собранного от этого источника;
- server OpenClaw core имеет собственный такой же явный контракт;
- tar-sync исчезает как основной live-механизм;
- server version определяется по manifest/commit/tag без ручного расследования;
- rollback делается через предыдущий artifact/tag/ref;
- `env/state` управляются отдельно от code deploy.

### 9.3 Что целевая схема не должна делать

- не должна тянуть `dev` по умолчанию;
- не должна деплоить из случайной локальной рабочей копии;
- не должна считать tar-snapshot канонической версией;
- не должна смешивать server state с git-кодом;
- не должна оставлять сервер без однозначного указания deployed version.

## 10. Правила deploy/release

Обязательные правила:

1. Нельзя деплоить из случайной локальной рабочей копии.
2. Нельзя безусловно тянуть `dev` по умолчанию.
3. Deploy всегда должен иметь явный `source/ref`.
4. Feature-ветка не является live deploy source напрямую.
5. Server `env/state` не являются git-кодом.
6. Tar-sync не является канонической deploy-моделью.
7. После deploy должно быть понятно, какой `repo/branch/tag/commit` сейчас на сервере.
8. Cloudbot/agent-контур и `/opt/openclaw` нельзя считать одним и тем же repo/branch path.
9. Любой deploy artifact должен иметь доказуемое происхождение.
10. Если ref не указан явно, deploy не должен стартовать.

## 11. Как проверять, что в проде нужная версия

Минимально необходимая проверяемость после deploy:

1. На сервере должен существовать manifest deployed version:
   - `repo`
   - `ref`
   - `commit`
   - `deployed_at_msk`
   - `deployed_by`
   - `delivery_method` (`git`, `artifact`, `tar-sync`)
2. Для git-based контура должна быть доступна команда, эквивалентная:
   - `git rev-parse HEAD`
   - `git branch --show-current` или явный pinned detached ref
   - `git remote get-url origin`
3. Для tar-synced контура должен быть сохранен manifest снапшота.
4. Для `/opt/openclaw` должен быть свой отдельный manifest или git revision marker.
5. Health/smoke-check после deploy должен ссылаться не только на “сервис жив”, но и на конкретную deployed version.

Что важно:

- в текущей подтвержденной локально рабочей копии такой version contract не реализован;
- его нужно ввести как первичное исправление, а не оставлять “на потом”.

## 12. Ближайшие исправления

### Что нужно сделать быстро

1. Явно запретить deploy по умолчанию из `dev` во всех скриптах и runbook.
2. Ввести один временный integration branch `integration`.
3. Добавить обязательный deploy parameter `--ref` или эквивалентный явный выбор source.
4. Начать писать на сервер version manifest для каждого deploy.
5. Зафиксировать отдельным документом, что `/opt/openclaw` — это отдельный repo/контур.

### Что требует отдельной архитектурной задачи

1. Привести `main/dev` к реальным ролям или заменить их на новую branch-модель.
2. Убрать tar-sync как live-модель и заменить его versioned artifact delivery.
3. Формализовать cron/agent_commit контуры как code + schedule + manifest.
4. Развести Cloudbot code, OpenClaw core, server env/config/state и backups по отдельным контрактам.

### Какие 3–7 исправлений являются приоритетными

1. Прекратить implicit deploy из `dev`.
2. Назначить один временный канонический источник интеграции.
3. Добавить server-side version manifest.
4. Зафиксировать `/opt/openclaw` как отдельный deploy-контур.
5. Формализовать tar-sync как переходный artifact transport, а не source of truth.
6. Документировать server `env/state` как внешний runtime contract.

## 13. Открытые риски и техдолг

Даже после первичной починки останутся риски:

- часть live runtime все еще находится вне одного канонического repo;
- tar-sync как переходный механизм остается рискованным, даже если станет versioned;
- без CI/CD дисциплина ref/tag/deploy останется частично ручной;
- пока не проведена ревизия `main/dev`, старые ожидания команды могут конфликтовать с новой временной моделью;
- пока не оформлен контракт `/opt/openclaw`, нельзя считать deploy-путь полностью управляемым;
- без инвентаризации server paths, cron jobs и backup artifacts останется слепая зона в rollback и диагностике.

## 14. Краткий вывод

Текущая deploy/release реальность распределена между несколькими источниками:

- git-контуром cloudbot-кода;
- отдельным серверным repo `/opt/openclaw`;
- tar-sync снапшотами агентских каталогов;
- server-only `env/state`.

Поэтому текущая схема неуправляема не из-за отсутствия “красивого CI/CD”, а потому что нет одного зафиксированного deploy source, нет надежной ветки интеграции, нет artifact contract и нет простого способа определить live version на сервере.

Минимально правильная переходная схема:

- признать один канонический git-источник для cloudbot-кода;
- deploy только по явному `ref`;
- ввести один integration branch;
- фиксировать deployed version на сервере;
- считать tar-sync только transport-механизмом;
- считать server `env/state` отдельным внешним контрактом.
