# GitHub workflow для Codex

## Канонический git-контур

Перед любой задачей нужно исходить из следующей карты:

1. Основной инженерный git-репозиторий Cloudbot/OpenClo/Codex:
   `/Users/pro2kuror/Desktop/Cloudbot/engineer`
2. `origin` этого репозитория:
   `https://github.com/Johnnie13Walker/codex-base.git`
3. Архитектурный и runbook-контур:
   `/Users/pro2kuror/Desktop/Cloudbot/architect`
4. Server-only repo:
   `/opt/openclaw`

Правило:

- не спрашивать пользователя, где лежит git-репозиторий и существует ли он, если путь `projects/engineer` доступен;
- текущую рабочую ветку, `origin` и git-состояние нужно проверять локально командами `git -C /Users/pro2kuror/Desktop/Cloudbot/engineer ...`;
- вопросы про git допустимы только если этот путь недоступен или локальная конфигурация изменилась.

## Безопасная схема

1. Базовая ветка разработки: `dev`.
2. Любая автономная задача выполняется в отдельной feature-ветке от `dev`.
3. `main` не используется для прямых изменений.
4. Любой результат автономной работы проверяется через diff, тесты и PR.

## Именование веток

Рекомендуемый формат:

- `feature/health-check`
- `feature/telegram-watchdog`
- `feature/bitrix-sync`
- `feature/nightly-report`

## Источники задач

Codex должен брать задачи только из одного источника:

1. `docs/PLAN.md`
2. Или явно указанного списка GitHub issues
3. Или контекста конкретного PR

Нельзя смешивать несколько backlog в одном автономном запуске без прямого указания пользователя.

## Что хорошо подходит для ночного запуска

- тесты и фиксы по ним;
- локальные рефакторинги;
- документация;
- health-check сценарии;
- логирование;
- наблюдаемость;
- безопасные интеграционные проверки;
- работа с issue и TODO-листами.

## Что нельзя делать автоматически без явного указания

- deploy;
- опасные миграции;
- удаление данных;
- ротация секретов;
- изменение production-конфигурации;
- массовые архитектурные перестройки.

## Пример задачи для GitHub / Codex

```text
Возьми первую незавершённую задачу из docs/PLAN.md, выполни её в отдельной feature-ветке от dev, запусти проверки, обнови docs/STATUS.md и подготовь изменения к PR без deploy.
```
