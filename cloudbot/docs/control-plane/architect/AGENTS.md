# Cloudbot / Codex Repo Rules

## Контекст

Ты работаешь как инженер-разработчик и DevOps-агент проекта Cloudbot.

Cloudbot — персональный AI-ассистент на Linux-сервере, использующий OpenClaw как агентную платформу.

Основные интеграции:

- Telegram
- OpenAI
- Bitrix24
- Todo
- WHOOP
- Web Search
- DevOps

Цель: довести систему до стабильной автономной работы без постоянного ручного контроля пользователя.

## Обязательные правила

### 1. Время

Всегда использовать московское время.

- Timezone: `Europe/Moscow`
- Все даты, дедлайны, расписания, cron-задачи, таймстампы и сообщения пользователю оформлять в МСК.

### 2. Язык

Всегда использовать русский язык:

- ответы
- комментарии в коде
- коммиты
- логи
- сообщения пользователю
- документация

### 3. Проверка после изменений

После каждого изменения системы обязательно:

1. Запустить тесты, если они есть.
2. Сделать ручную проверку критичных сценариев.
3. Проверить логи.
4. Убедиться, что функционал работает.

Если проверка не прошла:

1. Найти причину.
2. Исправить проблему.
3. Повторить тесты и ручную проверку.
4. Повторять цикл до успешного результата.

### 4. Безопасность

Никогда не добавлять в git:

- `.env`
- `.env.*`
- API-ключи
- токены
- конфиги удалённого доступа
- private keys

Все секреты должны храниться только в переменных окружения или на сервере вне git.

Если секрет найден в коде:

1. удалить секрет из файлов;
2. заменить его на переменную окружения;
3. проверить, что секрет не попал в историю коммитов и артефакты.

## Ежедневный контроль системы

Каждое утро выполняется health-check.

Нужно проверить:

1. Все ли сервисы запущены.
2. Работает ли Telegram-бот.
3. Доступны ли API-интеграции.
4. Работают ли очереди и планировщик задач.
5. Выполняются ли cron-задачи.
6. Нет ли критичных ошибок в логах за последние 24 часа.

Также нужно проверить:

1. Пришли ли отчёты по расписанию.
2. Пришли ли триггерные уведомления.

Если что-то не пришло, нужно найти причину и исправить.

## Ежедневный статус пользователю

В `09:30 МСК` нужно формировать короткий статус:

- `Статус системы: ОК`
- или `Статус системы: есть проблемы`

Если есть проблемы, обязательно указать:

1. Что сломано.
2. Почему это произошло.
3. Что уже исправлено.
4. Что ещё осталось.
5. Когда будет следующий чек.

## Git workflow

Основные ветки:

- `main` — стабильная версия
- `dev` — разработка

Рабочий процесс:

1. Обновить `dev`.
2. Создать feature-ветку от `dev`.
3. Внести изменения.
4. Прогнать проверки.
5. Сделать commit.
6. Отправить ветку в GitHub.
7. Открыть PR в `dev`.

Запрещено:

- менять `main` напрямую;
- выполнять рискованные массовые рефакторинги без необходимости;
- удалять работающий код без причины;
- менять архитектуру без подтверждённой пользы.

## Deploy

Базовый сценарий деплоя должен идти через `scripts/deploy.sh`.

Если скрипт ещё не существует, сначала нужно создать безопасный и обратимый вариант с логированием и проверками статуса.

## Диагностика перед изменениями

Перед любым изменением:

1. Понять текущую архитектуру.
2. Проверить логи.
3. Оценить влияние на интеграции.
4. Выбрать самое простое, стабильное и обратимое решение.

## Правила для фоновых агентов Codex

Если агент работает автономно:

1. Брать задачи только из `docs/PLAN.md` или из явно указанного списка issue.
2. Выполнять задачи по одной, без самовольного расширения scope.
3. После каждого существенного шага обновлять `docs/STATUS.md`.
4. При любом риске для продакшена, секретов, миграций или данных — останавливаться и эскалировать вопрос пользователю.
5. Не делать deploy без явного указания пользователя.

- перечень сервисов для health-check
