# Workflow `personal_assistant`

## Назначение

`personal_assistant` — отдельный workflow для личных сценариев пользователя. Он подключает Ларису Ивановну к внешним источникам и собирает итоговый ответ без захода в CRM и sales-контур.

## Поддерживаемые сценарии MVP

- `day_briefing`;
- `add_calendar_event`;
- `search_request`.

## Базовый порядок работы

1. Принять intent пользователя и дату в `Europe/Moscow`.
2. Определить нужный режим workflow.
3. Подтянуть данные из соответствующих providers.
4. Передать нормализованный контекст в `larisa_assistant`.
5. Вернуть итог в формате Telegram-friendly brief или подтверждения действия.

## Режим `day_briefing`

Источники:

- `calendar`;
- `todo`;
- `weather`;
- `news`.

Результат:

- встречи;
- задачи;
- свободные окна;
- погода по Москве;
- новости по выбранным темам;
- `3` действия на день.

## Режим `add_calendar_event`

Источники:

- `calendar`.

Результат:

- проверка обязательных полей;
- создание события;
- подтверждение пользователю;
- явное сообщение об ошибке, если событие не создано.

## Режим `search_request`

Источники:

- `search`.

Результат:

- краткий ответ по запросу;
- релевантные ссылки или выводы;
- явное сообщение об ограничении, если поиск недоступен.

## Границы workflow

- не маршрутизировать запросы в `sales_brief`;
- не дергать CRM как источник данных личного ассистента;
- не смешивать brief дня с коммерческими отчётами;
- не брать на себя роль глобального orchestrator.
