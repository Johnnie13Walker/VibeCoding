# Журнал решений

## Контур и границы

- создан отдельный контур `commercial-director`;
- выполнен role split от `main`;
- sales-контекст не смешивается с personal assistant;
- sales-контекст не смешивается с DevOps;
- sales-контекст не смешивается с CFO;
- структура папки ведётся отдельно;
- перенос делается через controlled extraction.

## Решения по содержанию

- в роль переносятся только материалы, напрямую относящиеся к продажам, воронке, дисциплине менеджеров и Telegram-отчётам;
- релевантными считаются: `Sales Copilot`, `Фокус РОПа`, `Bitrix24` sales-сущности, playbooks по pipeline, prompts и шаблоны;
- нерелевантными считаются: founder memory, personal assistant, DevOps-runbooks, delivery-layer Telegram, CFO-аналитика и общий infra-контур;
- специально не переносились runtime-код, bindings, scheduler, shell-обвязка и live-конфиги;
- дубли между legacy-блоками и новым knowledge-слоем отброшены в пользу одного рабочего шаблона на каждый блок.

## Основные шаблоны

Основными шаблонами роли считаются:

- `templates/telegram-report-full.md`;
- `templates/telegram-report-short.md`;
- `templates/block-funnel.md`;
- `templates/block-overdues.md`;
- `templates/block-rop-focus.md`.

`templates/block-money.md` оставлен только как архивный reference для сверки legacy-формата и не считается каноническим daily-шаблоном.

## Решения по отчётам

- `Воронка в работе` зафиксирована единым каноническим daily-блоком;
- отдельный daily-блок `Деньги` выведен из канонического формата и оставлен только как архивный deep-dive;
- `Просрочки по менеджерам` вынесены в отдельный блок;
- `Фокус РОПа` сделан коротким;
- `Фокус РОПа` не дублирует весь список просрочек;
- длинные названия задач режутся аккуратно;
- в `Просрочках` не выводятся менеджеры с нулём;
- при отсутствии подтверждённого источника ограничение пишется прямо в нужном блоке.
- `Sales Copilot` жёстко закреплён на каждый рабочий день в `09:30` по `Europe/Moscow` за предыдущий рабочий день;
- еженедельный отчёт по продажам жёстко закреплён на пятницу `18:30` по `Europe/Moscow` за рабочую неделю.

## Что считается релевантным для роли

- сделки и их движение;
- `следующий шаг`;
- стадии и риск сделки;
- коммуникации по сделкам;
- задачи, просрочки и переносы сроков;
- manager-level discipline;
- приоритеты для РОПа;
- структура и формат Telegram-отчёта.

## Что считается нерелевантным

- личные задачи основателя;
- общая админская и серверная эксплуатация;
- runtime config, bindings и cron как предмет текущего шага;
- управленческий финконтур вне продаж;
- не-sales рабочая память из `main`.

## Что не переносили специально

- live deploy;
- live routing;
- cron;
- bindings;
- runtime config;
- Telegram account setup;
- shared provider-слой;
- shell-автоматизацию `main`.

## Какие дубли отброшены

- дубль между `Воронкой в работе` и legacy-денежным блоком;
- дубль между `Просрочками по менеджерам` и `Фокусом РОПа`;
- дубль между новым knowledge-слоем и legacy-описаниями там, где достаточно одного шаблона;
- дубль alias-логики `/pipeline` и `/sales` внутри самого контура роли.

## Что требует ручного решения

- отдельный runtime роли;
- fixed routing;
- отказ от `channel=last`;
- отдельный agent routing;
- production `cron` под уже зафиксированное расписание;
- live-подтверждение `Bitrix24`, задач и коммуникаций;
- граница shared-layer между `main` и отдельной ролью.

## Открытые риски

- часть логики всё ещё живёт только в каноническом runtime `../../assistant/engineer`;
- без live `Bitrix24` часть интеграционных документов остаётся каркасом;
- пока не подтверждён доступ к задачам, нельзя считать блоки просрочек и `Фокус РОПа` production-ready;
- пока не подтверждён routing, этот контур остаётся knowledge/workspace-слоем, а не отдельным live-агентом.
