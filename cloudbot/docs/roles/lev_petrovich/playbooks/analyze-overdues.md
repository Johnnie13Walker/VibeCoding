# Плейбук: как анализировать просрочки по менеджерам

## Назначение

Показать, у каких менеджеров есть подтверждённые просрочки по задачам продаж и где проблема уже влияет на управляемость сделки.

## Какие данные нужны

- источник задач: `tasks.task.list` или другой подтверждённый рабочий источник;
- привязка задачи к сделке;
- статус задачи;
- дедлайн задачи;
- ответственный по задаче или по сделке;
- название сделки;
- при необходимости сумма и стадия сделки для приоритизации.

## Как определить значимую просрочку

- базовая просрочка: открытая задача с дедлайном в прошлом;
- значимая просрочка: просрочка, подтверждённая по задаче, привязанной к активной сделке;
- приоритет усиливается, если просрочка длится `3+` дня, висит на крупной сумме, относится к поздней стадии или срок переносили более `2` раз.

## Как агрегировать по менеджерам

- считать только подтверждённые просрочки;
- группировать по менеджеру сделки;
- для каждого менеджера считать:
  - количество просрочек;
  - максимальную просрочку;
  - `1-3` примера;
- менеджеров с `0` не выводить.

## Что выводить в отчёт

- отдельный блок `🗂 Просрочки по менеджерам`;
- по каждому менеджеру:
  - имя;
  - количество;
  - максимальная просрочка;
  - `1-3` коротких примера;
- если просрочек нет, писать это явно.

## Как не перегружать блок

- не выводить все задачи подряд;
- не давать больше `3` примеров на менеджера;
- не превращать блок в детальный лог задач;
- деталь держать только здесь, а не размазывать по всему отчёту.

## Какие сигналы особенно важны

- много просрочек у одного менеджера;
- длинная максимальная просрочка;
- просрочки на поздних стадиях;
- просрочки на крупных сделках;
- повторяющаяся проблема у одного и того же менеджера.

## Что делать, если данных не хватает

- не придумывать просрочки;
- не считать неподтверждённые привязки задачи к сделке;
- если нет рабочего доступа к задачам, прямо писать ограничение;
- в `Фокус РОПа` не тащить весь список просрочек, если нет данных по другим управленческим сигналам.

## Рабочее правило

- только менеджеры с реальными просрочками;
- нули не выводить;
- деталь держать в отдельном блоке;
- в `Фокус РОПа` переносить только короткий приоритет, а не весь список просрочек.
- если просрочка попадает в `Фокус РОПа`, она должна быть показана как управленческий кейс, а не как копия списка по менеджерам.
