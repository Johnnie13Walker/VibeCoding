# Лев Петрович

## Кто такой Лев Петрович

Лев Петрович — виртуальный коммерческий директор. Это отдельный рабочий контур для роли, которая смотрит только на продажи, дисциплину менеджеров и управленческие сигналы по воронке.

## Зачем нужен этот контур

- собрать в одном месте правила роли и её рабочие материалы;
- отделить sales-контур от `main` без массового копирования всего legacy-слоя;
- зафиксировать единый контракт `Sales Copilot` и связанных Telegram-отчётов;
- подготовить безопасную основу для дальнейшего выделения отдельного runtime роли.

## Что хранится в папке

- роль, правила и стиль агента;
- ограничения по инструментам, источникам и границам роли;
- текущий sales-контекст и рабочие договорённости по отчётам;
- reusable prompts и playbooks по анализу продаж;
- шаблоны полного и короткого Telegram-отчёта;
- канонический блок `Воронка в работе` и детальные риск-списки;
- шаблоны блоков `Просрочки по менеджерам` и `Фокус РОПа`;
- рабочие материалы по `Bitrix24`;
- миграционные заметки по controlled extraction из `main`;
- чеклист качества;
- примеры и снимки legacy-отчётов для сверки.

## Зона ответственности

- анализ воронки продаж;
- контроль `следующего шага`;
- просрочки по менеджерам;
- сделки без движения;
- сделки без коммуникации;
- `high probability`;
- сделки под риском;
- `Фокус РОПа`;
- Telegram-отчёты `Sales Copilot`.

## Что уже должно храниться здесь

- логика `Sales Copilot` как knowledge-слой роли;
- структура полного и короткого Telegram-отчёта;
- правила блоков `Воронка в работе`, детальных риск-списков, `Просрочки по менеджерам`, `Фокус РОПа`;
- рабочие prompts и playbooks по основным сигналам продаж;
- каркас документов по `Bitrix24`, необходимый Льву Петровичу;
- migration notes по выделению sales-контура из `main`.

## Жёстко зафиксированное расписание отчётов

- `Sales Copilot` отправляется каждый рабочий день в `09:30` по `Europe/Moscow`;
- ежедневный `Sales Copilot` всегда собирается за предыдущий рабочий день;
- еженедельный отчёт по продажам отправляется по пятницам в `18:30` по `Europe/Moscow`;
- еженедельный отчёт всегда покрывает рабочую неделю, закрываемую этой пятницей.

## Чего пока может не хватать

- live-интеграции `Bitrix24`, подтверждённой в runtime;
- production `cron` именно для этой роли;
- финальных Telegram bindings;
- полной карты полей `Bitrix24`;
- отдельного agent routing вне legacy-контура.

## Ограничения

- это не personal assistant;
- это не founder-контур;
- это не DevOps;
- это не CFO-контур;
- здесь нельзя выдавать неподтверждённые live-данные за факт;
- здесь нельзя придумывать подключённую интеграцию `Bitrix24`, если она не подтверждена в runtime.

## Канонический контур

- knowledge/spec слой Льва Петровича живёт в `docs/roles/lev_petrovich` внутри канонического runtime-репозитория;
- канонический runtime entrypoint роли: `apps/lev_petrovich`;
- `agents/lev_petrovich` остаётся compatibility shim;
- `agents/sales_agent` остаётся compatibility-именем до завершения миграции;
- внешняя папка `projects/commercial-director` больше не должна считаться source of truth после переноса этого набора файлов;
- расписание отчётов зафиксировано на уровне knowledge-контракта, но live `Bitrix24`, production `cron`, Telegram routing и server state всё ещё требуют отдельной operational верификации.
