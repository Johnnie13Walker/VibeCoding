# Runbook: Incident Response

## Когда запускать
- Сервис недоступен пользователям.
- Резко выросли ошибки 5xx/таймауты.
- Критичные алерты по инфраструктуре.

## Быстрые действия (первые 15 минут)
1. Зафиксировать время начала инцидента и затронутые компоненты.
2. Проверить доступность:
   - `curl -I https://your-domain/health`
   - `systemctl status <service>`
3. Снизить impact:
   - rollback на предыдущую версию,
   - restart упавшего сервиса,
   - переключение на fallback.
4. Обновить статус в командном канале.

## Диагностика
- Логи:
  - `journalctl -u <service> -n 200 --no-pager`
- Ресурсы:
  - `df -h`, `free -h`, `uptime`
- Сеть:
  - `ss -tuln`
- Ошибки приложения:
  - проверить rate ошибок и последние deploy changes.

## Завершение
1. Подтвердить восстановление метрик и доступности.
2. Закрыть инцидент только после периода наблюдения (15-30 минут).
3. Оформить postmortem:
   - первопричина,
   - влияние,
   - timeline,
   - preventive actions с владельцами и сроками.
