# Sales Copilot для Belberry

Sales Copilot в Cloudbot — это короткий управленческий контур для владельца или руководителя продаж. Его задача не показать “ещё один CRM-отчёт”, а быстро ответить на три вопроса:

- где деньги;
- где риск;
- куда сегодня вмешаться лично.

## Команды

- `/sales` — главный утренний отчёт Sales Copilot. Возвращает пакет из 3 отдельных Telegram-сообщений в строгом порядке: основной `sales`, затем `risks`, затем `focus`.
- `/pipeline` — alias на `/sales` для обратной совместимости.
- `/risks` — отдельный отчёт `⚠️ Риски по сделкам`.
- `/focus-sales` — отдельный отчёт `🎯 Фокус РОПа`.
- `/bitrixcheck` — диагностика Bitrix integration.

## Структура отчётов

### `/sales`

Формат morning dispatch:

- `1.` основная сводка Sales Copilot;
- `2.` отдельное follow-up сообщение `⚠️ Риски по сделкам`;
- `3.` отдельное follow-up сообщение `🎯 Фокус РОПа`.

Внутри основного `sales`:

- `📊 Sales Copilot`
- `🧾 Поток за вчера`;
- `📈 Состояние воронки на сейчас`;
- внутри этого же блока отдельно показывается горячая стадия `Подготовка договора` с точным перечнем сделок-ссылок на Bitrix24;
- отдельный блок `📞 Коммуникации за вчера`;
- отдельные блоки по встречам, брифам, отказам и запущенным сделкам;
- отдельный блок `ℹ️ Ограничение системы`, если есть ограничения по open lines или sales-фильтру подразделений.
- если в блоке упомянута сделка или лид, название выводится как HTML-ссылка на карточку Bitrix24;
- рядом всегда показывается ответственный менеджер, fallback — `без ответственного`.

Цель: прочитать за 10 секунд и сразу понять, где деньги и где нужен руководитель.

### `/pipeline`

`/pipeline` больше не живёт как отдельный отчёт.

Теперь:

- весь контур воронки собран внутри `/sales`;
- горячая стадия `Подготовка договора` выводится внутри блока `📈 Состояние воронки на сейчас`;
- команда `/pipeline` оставлена только как alias на `/sales` для обратной совместимости.

### `/risks`

Формат:

- `⚠️ Риски по сделкам`
- отдельный risk-report без `sales` и без `focus`
- используется как самостоятельная команда и как обязательный follow-up внутри `/sales`

### `/focus-sales`

Формат:

- `🎯 Фокус РОПа`
- до 7 сделок под контролем;
- короткие manager-level сигналы по дисциплине;
- верхнеуровневые проценты по отделу;
- 3–7 конкретных действий для РОПа на сегодня.

`/focus-sales` — отдельный отчёт РОПа. Он всегда идёт третьим сообщением внутри `/sales`, но может вызываться и отдельно.

## Откуда берутся данные

Используются только live-методы, которые доступны текущему Bitrix webhook:

- `profile`
- `user.get`
- `crm.deal.list`
- `crm.lead.list`
- `crm.company.list`
- `crm.contact.list`
- `calendar.event.get`
- `department.get`

Пока не используются как обязательный источник:

- `tasks.task.list`

## Как строятся ссылки и менеджеры

- Telegram-отчёты Sales Copilot отправляются с `parse_mode=HTML`, чтобы ссылки были кликабельными.
- Ссылка на сделку собирается по схеме `/crm/deal/details/{ID}/`.
- Ссылка на лид собирается по схеме `/crm/lead/details/{ID}/`.
- Домен берётся из Bitrix portal base URL.
- Если в install/handler state Bitrix лежит OAuth-host (`oauth.bitrix24.tech` / `oauth.bitrix.info`), Sales Copilot не использует его для карточек и берёт клиентский портал из `client_endpoint`.
- Для имени менеджера используются поля `ASSIGNED_BY_ID` у сделки или лида и справочник `user.get`.
- Для названия используются только исходные `TITLE`/`NAME` из CRM без автосклейки компании или контакта.
- Для лида в гиперссылке используется только короткое название лида без имени компании или контакта.
- Если название слишком длинное, Cloudbot сокращает его до читаемого вида и сохраняет кликабельность.

## Как читаются причины отказа

- Блок `Отказы за предыдущий рабочий день` не ограничивается стандартными `COMMENTS` / `DESCRIPTION`.
- Cloudbot дополнительно запрашивает `crm.deal.fields` и ищет Bitrix-поля отказа среди `UF_*`, если их label содержит `Причина отказа`, `Причины отказа` или `reason`.
- Если причина хранится как enum-код, Cloudbot расшифровывает ID в человекочитаемое значение через metadata поля.
- Если причина хранится как текст в кастомном поле, в отчёт уходит сам текст.
- Внутренние задачи и служебные stage-поля не считаются причиной отказа.
- Fallback `причина не указана в CRM` используется только если ни в dedicated refusal fields, ни в текстовых комментариях сделки причина не найдена.

## Коммуникации за вчера

В текущем MVP метрики считаются только по тем данным, которые реально доступны текущему webhook.

### Фильтр продажной команды

- В production source of truth для sales-структуры закреплён корневой отдел `Отдел продаж` (`department_id = 5`).
- Cloudbot берёт этот root и автоматически расширяет его на все дочерние sales-подразделения, чтобы новые группы и новые сотрудники попадали в отчёт без ручного дописывания.
- В текущей структуре это включает:
  `Группа продаж Belberry`, `Группа продаж Acoola Team`, `Телемаркетинг`.
- Приоритет конфига:
  `SALES_DEPARTMENT_IDS` -> `SALES_DEPARTMENT_NAMES`.
- Если `SALES_DEPARTMENT_IDS` заданы, используются именно они.
- Если ID не заданы, Cloudbot ищет точные совпадения по `department.get` и, когда найденные sales-группы являются siblings одного родителя продаж, поднимает фильтр до этого общего sales-root.
- После нахождения root sales-подразделений Cloudbot сверяется со структурой отдела и включает их дочерние узлы, если они появятся.
- Если точного совпадения нет, Cloudbot не падает: пишет warning и не подмешивает посторонние отделы.
- Сотрудники собираются через полный `user.get`, а не только по тем, кто засветился в сделках/встречах.
- В итоговый allowlist попадают только активные сотрудники этих sales-subdepartments.
- Этот же allowlist теперь применяется ко всему `Sales Copilot`, а не только к блоку коммуникаций:
  активные/закрытые сделки, просрочки, встречи, брифы и manager-level агрегаты режутся по подтверждённому sales-составу.
- Если ответственный не входит в allowlist, сущность не попадает в отчёт даже при наличии активности в CRM.
- Руководители отдела/групп и служебные сущности вида `Отдел продаж` из операционного расчёта исключаются.
- Для service/integration аккаунтов есть явный blacklist:
  `SALES_EXCLUDED_USER_IDS`, `SALES_EXCLUDED_USER_NAMES`, `SALES_EXCLUDED_USER_MARKERS`.
- По умолчанию Cloudbot дополнительно отсеивает аккаунты с явными служебными маркерами вроде `интегратор`, `integration`, `service account`, `bot`.
- Сотрудники финансов, рекрутинга, партнеров и других отделов в блоки `📞 Коммуникации за вчера` и `⚠️ По менеджерам` не попадают.

### Наборы

- Источник: `crm.activity.list`
- Используются записи с `TYPE_ID = 2`
- Для реальных звонков дополнительно проверяется `PROVIDER_ID = VOXIMPLANT_CALL` или `PROVIDER_TYPE_ID = CALL`
- Один call activity = один набор
- Дата звонка считается по `START_TIME`, fallback на `CREATED`

### Нормальные звонки

- Источник: `crm.activity.list`
- Нормальный звонок = длительность `>= 60 секунд`
- Длительность считается по разнице `END_TIME - START_TIME`
- Если Bitrix не отдаёт обе метки времени, звонок не считается “нормальным” автоматически

### Открытые линии

- Прямой источник должен быть `imopenlines.session.history.get`
- Если метод доступен, Cloudbot считает коммуникации только по операторам из sales-allowlist
- Если метод недоступен, Cloudbot не выдумывает метрику и показывает системное ограничение `Открытые линии недоступны`

### Управленческие сигналы

- много наборов, но мало разговоров;
- низкая коммуникационная активность;
- отсутствие коммуникаций у менеджера за вчера.

### Формат блока `📞 Коммуникации за предыдущий рабочий день`

- блок выводится не списком и не markdown-таблицей, а в compact pseudo-table внутри Telegram `preformatted` блока;
- строки выравниваются моноширинно, чтобы руководитель сразу видел `Статус`, `Менеджер`, `Роль`, `Наб`, `Дзв`, `Конв`, `Чаты`, `Встр`, `Опер`;
- столбец `Роль` обязателен:
  `TM` — телемаркетинг;
  `SM` — менеджер по продажам;
- длинные имена режутся до фиксированной ширины и заканчиваются на `...`;
- в итоговом блоке сотрудники сортируются от сильных к слабым, чтобы сверху сразу были лидеры дня, а ниже — просадка;
- в агрегате по отделу считаются:
  `Наб`, `Дзв`, агрегированная `Конв`, `Чаты`, `Встр` только по sales-менеджерам и средний `Опер`;
- после таблицы всегда идёт одна короткая управленческая строка-вывод.

### Role-based модель `Опер`

`Опер` считается не через конверсию, а через активные клиентские минуты за день.

Целевая загрузка: `300` минут чистой клиентской работы за день.

Общая формула:

- `Опер = min(10; Активные минуты / 300 × 10)`.
- `Конв` остаётся в отчёте как справочная метрика качества, но в рейтинг `Опер` не входит.
- `Чаты` считаются как уникальные диалоги в мессенджерах за день, а не как количество сообщений.

Для менеджеров продаж (`Группа продаж Belberry`, `Группа продаж Acoola Team`) score строится из:

- пустых наборов;
- дозвонов;
- диалогов в мессенджерах;
- встреч.

Правила:

- `Активные минуты = min((Наб - Дзв) × 1.5; 90) + Дзв × 15 + Чаты × 10 + Встр × 50`;
- встреча входит в score только для sales-менеджеров;
- встречи матчятся по `assigned_id`, а не по строке ФИО, чтобы короткое имя из Bitrix не обнуляло `Встр`;
- пустые наборы ограничены сверху через `min(...; 90)`, чтобы нельзя было накрутить рейтинг.

Для телемаркетинга (`Телемаркетинг`) score строится из:

- пустых наборов;
- дозвонов;
- диалогов в мессенджерах.

Правила:

- `Активные минуты = min((Наб - Дзв) × 1.5; 90) + Дзв × 12 + Чаты × 10`;
- встречи для телемаркетинга не учитываются вообще;
- пустые наборы ограничены сверху через `min(...; 90)`, чтобы нельзя было накрутить рейтинг.

### Статусы блока

- `🔥 НОРМ` — нормальная вовлечённость по role-based модели;
- `⚠️ РИСК` — активность есть, но она слабая или недостаточная;
- `❌ СТОП` — meaningful-активности почти нет или есть имитация работы без разговоров.

## Как формируется `🎯 Фокус РОПа`

Блок объединяет старые `Риски по продажам` и `Фокус продаж`, но живёт только в отдельном отчёте `/focus-sales`, чтобы одна и та же сделка не дублировалась в `/sales`, `/pipeline` и других отчётах.

Внутри него всегда 4 секции:

### `📌 Сделки под контролем`

До 7 сделок по единому приоритету:

1. сумма сделки;
2. поздняя стадия;
3. давняя тишина по коммуникациям;
4. отсутствие следующего шага;
5. риск потери;
6. признак `needs_leader`.

Каждая строка показывает:

- ссылку на сделку;
- менеджера;
- сумму;
- стадию;
- одну короткую причину, почему сделка в фокусе.

### `👥 Команда и дисциплина`

Показываются только активные сотрудники из 3 sales-подразделений:

- `Группа продаж Belberry`
- `Группа продаж Acoola Team`
- `Телемаркетинг`

Сигналы по менеджерам берутся по приоритету:

1. просроченные задачи по сделкам;
2. нет активности за вчера;
3. много наборов, мало разговоров;
4. много сделок без следующего шага;
5. есть запущенные сделки без коммуникации;
6. много отказов за вчера;
7. поздние стадии без движения;
8. низкая коммуникационная активность.

### `📊 Сигналы по отделу`

Для РОПа считаются:

- `% сделок без следующего шага`
- `% сделок без движения`
- `% запущенных сделок`
- `% менеджеров с просроченными задачами`
- `% менеджеров со слабой коммуникационной активностью`
- число менеджеров без активности за вчера
- число и сумма отказов за вчера

### `➡️ Что сделать РОПу сегодня`

Формируются конкретные действия по данным:

- разобрать просроченные задачи;
- вернуть в контакт запущенную сделку;
- проверить сделки без следующего шага;
- разобрать свежие отказы;
- проверить движение по самой приоритетной сделке.

## Эвристики

Текущий MVP использует простые управленческие правила:

- нет следующего шага = риск;
- нет активности больше 5 дней = риск;
- поздняя стадия + нет активности = высокий риск;
- крупная сделка = высокий приоритет;
- сделка с сегодняшней встречей = повышенное внимание;
- крупная или поздняя сделка с риском = кандидат на личный контроль руководителя.

Для `/focus-sales` приоритет строится в таком порядке:

1. сумма сделки;
2. поздняя стадия;
3. давняя тишина;
4. нет следующего шага;
5. нужен руководитель.

## Важные ограничения данных

- Для сделок `следующий шаг` берётся в первую очередь из `crm.activity.list`.
- Если в `crm.activity.list` шага нет, Sales Copilot дополнительно проверяет открытые встречи из смарт-процесса встреч и подтягивает их как `next step`, если встреча связана со сделкой через `parentId2`.
- Внутренняя задача Bitrix может быть `next step`, но сама по себе не считается клиентской коммуникацией.
- Блок `без коммуникации > 14 дн.` опирается на подтверждённое клиентское касание (`LAST_COMMUNICATION_TIME` / звонковые activity), а также на живую назначенную встречу по сделке; внутренние задачи менеджера этот сигнал не обнуляют.
- `tasks.task.list` сейчас доступен по `app_oauth` и используется как дополнительный контекст, но не подменяет факт коммуникации с клиентом.
- Названия стадий сделок берутся из доступного `STAGE_ID`; если Bitrix не отдаёт читаемое имя стадии, Cloudbot показывает нормализованное техническое название.
- Если ответственный по ID не разрешился через `user.get`, отчёт показывает `без ответственного`, но сама карточка сущности остаётся кликабельной.
- `crm.activity.list` доступен и используется как основной live-источник звонков.
- Фильтр продажной команды строится не по одному “общему отделу продаж”, а по трем точным подподразделениям: `Группа продаж Belberry`, `Группа продаж Acoola Team`, `Телемаркетинг`.
- Если хотя бы одно из этих подразделений не найдено по точному совпадению или если сотрудников по `UF_DEPARTMENT` собрать не удалось, Cloudbot пишет warning и не смешивает их с другими отделами.
- Прямой telephony-метод `voximplant.statistic.get` callable, но в проверке за вчера не дал данных, поэтому не используется как primary source.
- `telephony.statistic.get` недоступен: `method not found`.
- Прямой метод open lines `imopenlines.session.history.get` недоступен по scope: `access denied`.
- В activity-layer `TYPE_ID = 4` в проверке соответствовал email-активностям (`CRM_EMAIL`), а не open lines, поэтому email не подменяет метрику открытых линий.

## Автоматические задания

- `09:30 МСК` — `sales_brief`
- сразу после `sales_brief` отдельным сообщением уходит `Фокус РОПа`
- `17:00 МСК` — `sales_followup`
- `Пятница 18:30 МСК` — `sales_weekly_review`
- delivery sales-отчётов идёт в fixed target `SALES_TELEGRAM_CHAT_ID`
- если `SALES_TELEGRAM_CHAT_ID` не задан, daily и follow-up берут `SALES_WEEKLY_TELEGRAM_CHAT_ID`
- weekly сначала берёт `SALES_WEEKLY_TELEGRAM_CHAT_ID`, потом `SALES_TELEGRAM_CHAT_ID`
