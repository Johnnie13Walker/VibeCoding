# Daily Health Check Prompt

Используй этот текст для ежедневного запуска Codex в `09:30 МСК`.

```text
Выполни утренний health-check Cloudbot.

Правила:
- Русский язык.
- Все времена указывай в Europe/Moscow.
- Никакого deploy.
- Не меняй код, если достаточно диагностировать проблему и зафиксировать статус.
- Если исправление безопасное и локальное, можно внести его, затем обязательно прогнать проверки.

Правила интерпретации:
- Считай проблемой только текущий или неустранённый сбой.
- Перед эскалацией cron/job как активной проблемы сначала перепроверь самый свежий run именно этого job и сравни времена последних запусков.
- Единичный `error` у cron/job не считай активной поломкой, если следующий запуск того же job в пределах 10 минут завершился `ok`.
- Если после `error` уже есть более свежий `ok` или `delivered`, считай это self-healed случаем и выноси в `Наблюдения/техдолг`, а не в активные проблемы.
- Если свежий run этого job ещё `running`, не объявляй старый `error` активной поломкой до завершения текущего run; сначала зафиксируй, что проверка в процессе, и укажи ближайший повторный чек.
- `missing scope: operator.read` в `openclaw gateway probe` или `openclaw status --deep` трактуй как деградированную глубину диагностики, а не как недоступность gateway, если базовая reachability в норме.
- `multi_user_heuristic` трактуй как архитектурный риск/техдолг, а не как активную поломку, если используются allowlist-политики и нет фактических сбоев gateway, Telegram, cron или доставки.
- Доступное обновление OpenClaw без деградации сервиса считай maintenance backlog, а не аварией.
- Не выдавай `API OK` за `пользовательский сценарий OK`: статус интеграции, статус skill и status user-facing capability должны интерпретироваться отдельно.
- Если интеграция давно известна и подтверждена, не выводи её в блоке `Новые API`; такие интеграции должны жить в основном блоке `API и интеграции`.
- Для web search отдельно различай:
  - `Web Search provider`
  - `Web search skill`
  - `Web search для Ларисы`
  Эти статусы могут не совпадать.

Что проверить:
1. Все сервисы из проекта запущены.
2. Telegram-бот отвечает.
3. Внешние API-интеграции доступны.
4. Очереди и планировщик работают.
5. Cron-задачи выполнялись.
6. Нет критичных ошибок в логах за последние 24 часа.
7. Ежедневные отчёты и триггерные уведомления реально приходили.
8. Все известные API отражены в основном блоке интеграций.
9. Пользовательские возможности отражены отдельно от API.

После проверки:
1. Обнови docs/STATUS.md.
2. Сформируй ОДИН итоговый утренний отчёт пользователю, а не несколько отдельных сообщений.
3. Используй визуальный каркас в стиле `🟢 СЕРВЕР СТАБИЛЕН` / `🔴 ЕСТЬ ПРОБЛЕМЫ`.
4. В отчёте всегда покажи блоки:
   - `🧠 Итог`
   - `🖥 Хост`
   - `☁️ OpenClaw / OpenCloud`
   - `🔌 API и интеграции`
   - `🧩 Пользовательские возможности`
   - `⏱ Планировщик и доставка`
   - `🛡 Безопасность`
   - `📌 Наблюдения / техдолг`
   - `📊 Индекс здоровья`
   - `🕒 Время отчёта`
5. Блок `🆕 Новые API` показывай только если интеграция реально впервые обнаружена и ещё не переведена в штатный контур.
6. Блок `🚨 Активные проблемы` показывай только если есть текущие/неустранённые сбои. В нём укажи:
   - что сломано;
   - почему;
   - что уже исправлено;
   - что осталось;
   - когда будет следующий чек.
7. Если активных проблем нет, но есть деградированная диагностика или техдолг, вынеси их отдельным коротким блоком `📌 Наблюдения / техдолг`.
8. Полный read-only payload и промежуточные выводы не отправляй отдельными сообщениями при зелёном статусе; они должны оставаться внутренней диагностикой.
```
