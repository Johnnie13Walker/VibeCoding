# Контур агента «Финансист»

## 1. Назначение

`Финансист` — отдельный управленческий финансовый AI-контур для владельца digital-агентства.

Его задача:

- собирать финансовые данные из разных источников;
- приводить их к единой управленческой модели;
- делать прикладной анализ для собственника;
- выявлять риски, перекосы, кассовые проблемы и слабые места;
- выдавать жесткие, короткие и проверяемые выводы по цифрам.

Это не бухгалтерический справочник и не assistant-контур.  
Это управленческий аналитик бизнеса.

## 2. Граница ответственности

### За что отвечает

- Управленческий P&L / ОПиУ.
- ДДС / cashflow / кассовые разрывы.
- Структура постоянных и переменных расходов.
- ФОТ, нагрузка ФОТ на выручку, payroll pressure.
- Рентабельность по направлениям, клиентам, продуктам, юрлицам.
- Дебиторка, кредиторка, налоговая нагрузка.
- Сверка факт/план.
- Поиск аномалий, перекосов, несостыковок и подозрительных расходов.
- Регулярные финансовые сводки и разовые ad-hoc разборы.

### За что не отвечает

- Продажная воронка и CRM-аналитика как основной сценарий `Sales Copilot`.
- Личный ассистентный контур `Ларисы Ивановны`.
- Архитектурная приемка, deploy и runtime governance.
- Бухгалтерская консультация по нормам права.
- Выдумывание цифр при неполных данных.

## 3. Целевая архитектура контура

### 3.1 Product layer

Пользователь задает вопросы уровня собственника:

- что происходит с финансами сейчас;
- где течет маржа;
- какие расходы перегреты;
- где риск кассового разрыва;
- какие клиенты/направления тянут вниз;
- какие действия надо принять.

### 3.2 Agent layer

Отдельный контур:

- `agent_id`: `finansist`
- `display_name`: `Финансист`
- `role`: `Управленческий финансовый AI-агент`

Отдельные артефакты в инженерном контуре:

- `agents/finansist/agent.yaml`
- `agents/finansist/prompt.md`
- `agents/finansist/__init__.py`
- `agents/finansist/agent.py`
- `agents/finansist/telegram_route.py`
- `agents/finansist/workflows/*.py`
- `agents/finansist/providers/*.py`
- `agents/finansist/schemas/*.py`

### 3.3 Routing layer

Нужен отдельный route / intent / persona без смешения с другими агентами.

Рекомендуемая схема:

- `persona`: `finansist`
- `route namespace`: `finance`
- `primary command`: `/finance`
- дополнительные команды:
  - `/pnl`
  - `/cashflow`
  - `/runway`
  - `/expenses`
  - `/clients-profit`
  - `/ar`
  - `/ap`
  - `/finance-risks`
- базовые intents:
  - `finance_summary`
  - `pnl_analysis`
  - `cashflow_analysis`
  - `expense_structure_analysis`
  - `client_profitability_analysis`
  - `receivables_analysis`
  - `payables_analysis`
  - `finance_anomaly_scan`
  - `fact_plan_analysis`

### 3.4 Workflow layer

Канонические workflow контура:

1. `finance_summary`
2. `pnl_analysis`
3. `cashflow_analysis`
4. `expense_structure_analysis`
5. `profitability_analysis`
6. `receivables_analysis`
7. `client_profitability_analysis`
8. `fact_plan_analysis`
9. `finance_anomaly_scan`
10. `finance_weekly_report`
11. `finance_monthly_report`
12. `finance_board_brief`

Правило:

- один запрос пользователя должен попадать в один канонический workflow;
- дополнительные расчеты подключаются как skills/providers, а не дублируются в других агентах.

### 3.5 Data ingestion layer

Нужен отдельный слой загрузки данных с нормализацией, а не прямой анализ сырых таблиц.

Источники первого приоритета:

1. Google Sheets
2. Google Docs
3. Excel / `xlsx`
4. `csv`
5. Bitrix24

Целевая цепочка:

`source adapters -> normalization -> finance model -> analytics engine -> response formatter`

### 3.6 Normalized finance model

Ключевые сущности, которые нужны в первую очередь:

1. `period`
2. `legal_entity`
3. `business_unit`
4. `direction`
5. `product`
6. `client`
7. `project`
8. `manager` / `owner`
9. `revenue_item`
10. `expense_item`
11. `expense_category`
12. `fixed_cost`
13. `variable_cost`
14. `payroll_item`
15. `invoice`
16. `payment`
17. `cash_movement`
18. `receivable`
19. `payable`
20. `tax_obligation`
21. `plan_value`
22. `fact_value`

Обязательные измерения:

- месяц;
- юрлицо;
- клиент;
- направление;
- проект;
- статья дохода/расхода;
- ответственный;
- источник данных;
- дата документа;
- дата оплаты;
- статус сверки.

### 3.7 Analytics engine

Аналитический слой должен делать не только агрегаты, но и управленческие проверки:

- отклонение MoM;
- отклонение к плану;
- маржинальность по сегментам;
- рост расходов без роста выручки;
- рост ФОТ без роста валовой прибыли;
- концентрация выручки на 1-3 клиентах;
- долгие хвосты дебиторки;
- кассовый риск по неделям;
- аномалии статей затрат;
- несостыковки между P&L, ДДС и источниками;
- деградация unit-экономики.

### 3.8 Output layer

Формат ответа по умолчанию:

1. Суть
2. Что вижу по цифрам
3. Риски
4. Причины или гипотезы причин
5. Что делать
6. Какие данные еще нужны

Во всех ответах агент обязан разделять:

- `Факты`
- `Выводы`
- `Гипотезы`

## 4. Функции агента

`Финансист` должен уметь:

1. Загружать и связывать данные из нескольких источников.
2. Нормализовывать статьи, периоды, клиентов и юрлица.
3. Собирать управленческий P&L.
4. Собирать и анализировать ДДС.
5. Считать валовую, операционную и чистую прибыль.
6. Анализировать ФОТ и нагрузку ФОТ на выручку.
7. Искать перегретые и неэффективные статьи затрат.
8. Показывать рентабельность по направлениям и клиентам.
9. Анализировать дебиторку и кредиторку.
10. Выявлять риск кассового разрыва.
11. Сравнивать факт с планом.
12. Находить аномалии, перекосы и дырки в данных.
13. Формировать разовые ответы по вопросу собственника.
14. Формировать регулярные weekly/monthly/board summary отчеты.
15. Явно перечислять, каких данных не хватает для точного вывода.

## 5. Источники данных

### Базовые

1. Google Sheets с ОПиУ, ДДС, бюджетом, клиентской выручкой, ФОТ и планом.
2. Google Docs с финансовыми комментариями, управленческими заметками, планами, расшифровками и текстовыми пояснениями к цифрам.
3. Excel-файлы с финансовыми моделями, выгрузками и сверками.
4. Bitrix24:
   - сделки;
   - компании;
   - счета;
   - акты;
   - оплаты;
   - ответственные;
   - воронка как вспомогательный источник для выручки и дебиторки.
5. `csv`-выгрузки из банков, CRM, бухгалтерии и рекламных кабинетов.

### Второй приоритет

1. Банковские выписки.
2. Данные по налогам.
3. Табели/ФОТ/штатка.
4. Плановые бюджеты по направлениям.
5. Рекламные расходы по каналам.
6. Производственная себестоимость по проектам.

## 6. Базовые workflow

### Обязательный набор v1

1. `pnl_analysis`
   Назначение: собрать управленческий P&L за период и показать динамику.

2. `cashflow_analysis`
   Назначение: показать входящий/исходящий денежный поток, кассовые ямы и ближайшие риски.

3. `expense_structure_analysis`
   Назначение: разложить расходы по категориям, долям и динамике.

4. `profitability_analysis`
   Назначение: показать рентабельность по направлениям, продуктам и юрлицам.

5. `client_profitability_analysis`
   Назначение: выявить малорентабельных и убыточных клиентов.

6. `receivables_analysis`
   Назначение: показать дебиторку, просрочку и концентрацию риска.

7. `fact_plan_analysis`
   Назначение: сопоставить бюджет/план с фактом и выделить критичные отклонения.

8. `finance_anomaly_scan`
   Назначение: найти перегибы, всплески и несостыковки в цифрах.

### Регулярные workflow

1. `finance_weekly_report`
   Формат: короткий weekly owner brief по выручке, поступлениям, рискам и дебиторке.

2. `finance_monthly_report`
   Формат: месячный P&L + ДДС + структура расходов + риски + действия.

3. `finance_board_brief`
   Формат: сводка для собственника/управляющего партнера по текущему состоянию экономики бизнеса.

## 7. Базовые отчеты

Сразу стоит делать следующие отчеты:

1. P&L по месяцам.
2. ДДС по неделям и месяцам.
3. Структура расходов по блокам.
4. ФОТ и доля ФОТ в выручке.
5. Валовая прибыль по направлениям.
6. Операционная прибыль по направлениям.
7. Рентабельность по клиентам.
8. Дебиторка с aging.
9. Факт/план по выручке и расходам.
10. Список аномалий и тревожных сигналов.
11. Топ-5 перегретых статей затрат.
12. Топ-5 клиентов с худшей экономикой.

## 8. Обязательные метрики

1. Выручка.
2. Валовая прибыль.
3. Валовая маржа.
4. Операционная прибыль.
5. Операционная маржа.
6. Чистая прибыль.
7. EBITDA-like proxy, если амортизация несущественна и это явно оговорено.
8. Постоянные расходы.
9. Переменные расходы.
10. ФОТ.
11. Доля ФОТ в выручке.
12. CAC / cost-to-serve proxy, если доступны данные.
13. AR turnover / средний срок дебиторки.
14. AP turnover / средний срок кредиторки.
15. Cash runway.
16. Net cash gap по неделям.
17. Concentration ratio по выручке на топ-клиентов.
18. План-факт отклонение в абсолюте и процентах.
19. Revenue per employee, если доступна численность/FTE.
20. Gross profit per client / per project.

## 9. Риски данных и интеграций

Критичные риски, которые будут мешать качественному анализу:

1. Разные справочники клиентов и юрлиц между Google Sheets, Excel и Bitrix24.
2. Отсутствие единого `client_id` / `project_id`.
3. Смещение дат между отгрузкой, актом, оплатой и признанием выручки.
4. Смешение управленческого и бухгалтерского учета в одном источнике.
5. Неполный или грязный ФОТ.
6. Отсутствие аллокации общих расходов на направления и клиентов.
7. Нерегулярное обновление таблиц.
8. Ручные правки в Excel без audit trail.
9. Дубли строк при нескольких выгрузках.
10. Разные версии одного и того же отчета.
11. Неполные данные по налогам и обязательным платежам.
12. Непривязанные платежи к клиентам, проектам или статьям.
13. Невозможность reliable read из Bitrix24 по нужным сущностям.
14. Переопределение формул пользователем в Excel/Sheets.
15. Отсутствие плана, при котором план-факт анализ становится фикцией.

4. потом регулярная рассылка отчетов.

На первом этапе запрещено:

- писать обратно в источники;
- менять формулы пользователя без явного режима reconcile;
- смешивать finance-контур с assistant-контуром.

## 12. Типовые запросы к агенту

1. Покажи, что происходит с финансами за текущий месяц.
2. Где у нас сейчас самые слабые места по экономике бизнеса.
3. Какие статьи расходов перегреты за последние 3 месяца.
4. Где падает валовая маржа и на каких цифрах это видно.
5. Какие направления убыточны или близки к нулевой рентабельности.
6. Какие клиенты дают выручку, но съедают прибыль.
7. Где у нас риск кассового разрыва в ближайшие 4 недели.
8. Покажи дебиторку с разбивкой по срокам и зонам риска.
9. Сравни факт с планом по выручке и расходам за квартал.
10. Покажи, как изменился ФОТ и его доля в выручке.
11. Найди аномальные расходы, которые выросли без понятного результата.
12. Где есть несостыковки между ОПиУ, ДДС и оплатами.
13. Какие 5 управленческих действий дадут самый быстрый финансовый эффект.
14. Какие данные не дают тебе сделать точный вывод по рентабельности клиентов.
15. Подготовь короткую месячную сводку для собственника без бухгалтерской воды.

## 13. Пошаговый план запуска

### Этап 1. Контракт данных

1. Зафиксировать источники и владельцев данных.
2. Согласовать канонический справочник клиентов, юрлиц, направлений и статей.
3. Определить, что считается выручкой, валовой прибылью, OPEX, ФОТ и чистой прибылью именно в управленческой модели.
4. Зафиксировать правила признания выручки и оплат.

### Этап 2. Read-only интеграции

1. Подключить Google Sheets.
2. Подключить Google Docs.
3. Подключить Excel / `xlsx`.
4. Подключить `csv`.
5. Подключить Bitrix24 на чтение.
6. Сделать smoke-check по каждому источнику.

### Этап 3. Нормализация

1. Собрать единый слой маппинга сущностей.
2. Убрать дубли и разъезды имен.
3. Добавить признак качества и полноты данных.
4. Научить агент явно сообщать о конфликтующих источниках.

### Этап 4. Workflow v1

1. Запустить `pnl_analysis`.
2. Запустить `cashflow_analysis`.
3. Запустить `expense_structure_analysis`.
4. Запустить `client_profitability_analysis`.
5. Запустить `finance_anomaly_scan`.

### Этап 5. Регулярные отчеты

1. Еженедельный owner report.
2. Ежемесячный финансовый отчет.
3. Отдельный риск-репорт по кассе и дебиторке.

### Этап 6. Приемка

1. Проверить, что агент не фантазирует цифры.
2. Проверить, что агент отделяет факты от гипотез.
3. Проверить, что он не уходит в бухгалтерские лекции.
4. Проверить, что выводы опираются на цифры и отклонения.
5. Проверить, что на тестовых кейсах он называет проблемы прямо.

## 14. Критерии готовности v1

Контур `Финансист` можно считать запущенным, когда:

1. существует отдельный route / intent / persona;
2. доступны минимум Google Sheets, Excel и Bitrix24 read-only;
3. есть единая нормализованная модель выручки, расходов, оплат и обязательств;
4. работают 5 базовых workflow;
5. есть минимум 3 базовых owner-отчета;
6. агент явно пишет про нехватку данных;
7. ответы действительно управленческие, а не описательные.
