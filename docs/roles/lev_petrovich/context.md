# Контекст роли

## Назначение файла

Файл собирает текущий контекст роли Льва Петровича. На него должны опираться шаблоны, промпты, плейбуки и миграционные решения.

## Формат ежедневной сводки по продажам

Ежедневная сводка по продажам — это Telegram-отчёт Льва Петровича. Его задача: быстро показать, где деньги, где риск потери контроля и куда РОПу или руководителю нужно вмешаться сегодня.

## Жёстко зафиксированное расписание

- `Sales Copilot` отправляется каждый рабочий день в `09:30` по `Europe/Moscow`;
- daily-отчёт строится за предыдущий рабочий день, а не за текущий незавершённый день;
- еженедельный отчёт по продажам отправляется по пятницам в `18:30` по `Europe/Moscow`;
- weekly-отчёт строится за рабочую неделю, закрываемую этой пятницей.

## Ключевые договорённости по структуре

- блоки отчёта идут в фиксированном порядке;
- `Краткая сводка` остаётся короткой и не дублирует следующие блоки;
- `Воронка в работе` собирается единым блоком и уже содержит денежные строки там, где это нужно;
- отдельного ежедневного блока `Деньги` в канонической сводке больше нет;
- детальные списки `Сделки без коммуникации > 14 дн.` и `Сделки без следующего шага` идут после воронки как расшифровка риска;
- `Просрочки по менеджерам` живут отдельным блоком;
- `Фокус РОПа` короткий и не копирует блок просрочек;
- если источник не подтверждён, это указывается прямо внутри нужного блока.

## Блок «Краткая сводка»

Это верхний управленческий слой отчёта.

Смысл блока:

- показать `2-4` главных сигнала дня;
- не пересказывать целиком `Воронку в работе` и `Фокус РОПа`;
- выносить сюда только то, что реально меняет приоритет дня;
- если данные частично ограничены, коротко назвать ограничение здесь, а детали оставить в профильном блоке.

## Блок «Воронка в работе»

Это основной операционный блок ежедневной сводки. В нём используются только такие строки:

- `Активные сделки: количество / сумма`
- `С движением за последнюю неделю: количество`
- `Без движения за последнюю неделю: количество`
- `Этап договора: количество / сумма`
- `Под риском: количество / сумма`
- под `Под риском` только:
  - `Без следующего шага: количество / сумма`
  - `Без коммуникации > 14 дн.: количество / сумма`
  - `С просроченными задачами: количество / сумма`

Смысл блока:

- показать состав воронки;
- показать качество движения и текущий денежный вес активных сделок;
- показать, где теряется управляемость;
- не раскладывать рядом отдельный денежный блок с теми же агрегатами.

## Блок «Сделки без коммуникации > 14 дн.»

Это короткая детальная расшифровка самого жёсткого сигнала по тишине.

Правила блока:

- перечислять только подтверждённые сделки с длинной паузой;
- не повторять здесь всю воронку;
- по каждой строке нужны сделка, владелец, сумма и краткий риск;
- если таких сделок нет, это пишется явно;
- если источник коммуникаций не подтверждён, писать короткое ограничение вместо ложного нуля.

## Блок «Сделки без следующего шага»

Это детальная расшифровка сделок, которые потеряли понятный следующий ход.

Правила блока:

- перечислять только сделки без подтверждённого следующего действия;
- показывать сумму, владельца и короткий контекст риска;
- не превращать блок в повтор `Фокуса РОПа`;
- если сделок нет, это пишется явно;
- если поле `следующего шага` или proxy не подтверждены, блок помечается ограничением.

## Блок «Просрочки по менеджерам»

Правила блока:

- выводить только менеджеров с реальными просрочками;
- для каждого менеджера показывать имя;
- обязательно показывать количество просрочек;
- обязательно показывать максимальную просрочку;
- обязательно давать `1-3` примера;
- если просрочек нет, писать это явно.

Смысл блока:

- показать дисциплину менеджеров;
- вынести деталь по просрочкам в отдельный блок;
- не смешивать просрочки с `Фокусом РОПа`.

## Блок «Фокус РОПа»

Правила блока:

- это короткий управленческий блок;
- он не дублирует полный блок `Воронки в работе`;
- он не дублирует денежные строки и детальные риск-списки из соседних блоков;
- он не дублирует весь список просрочек;
- он не копирует таблицу коммуникаций менеджеров;
- он не копирует полные списки `Без следующего шага` и `Без коммуникации > 14 дн.`, если они уже есть в соседних блоках;
- в блоке допустимы только такие подпункты:
  - `Где потерян контроль`;
  - `💸 Где взять деньги сегодня`;
  - `☎️ Где РОПу самому связаться с клиентом`;
  - `🕳 Упущенные возможности`;
  - `Кого дожать сегодня`;
  - `Что РОП обязан сделать сегодня`;
  - `🔥 Молодцы за вчера`;
- `Где потерян контроль` показывает только `3-4` главных сигнала потери управляемости;
- `💸 Где взять деньги сегодня` ограничен `2-3` сделками с суммой и действием;
- `☎️ Где РОПу самому связаться с клиентом` ограничен `1-2` сделками и требует явной причины, почему РОП должен сам выйти к клиенту;
- `🕳 Упущенные возможности` показывает только провалы между обязательными этапами продажи, прежде всего между брифом и защитой КП;
- `Кого дожать сегодня` ограничен `2-3` сотрудниками с чётким действием на день;
- `Что РОП обязан сделать сегодня` ограничен `3` пунктами;
- `🔥 Молодцы за вчера` показывает только сотрудников с `9+/10` по готовой оценке `операционная вовлечённость за вчера`;
- ФИО менеджеров и сотрудников внутри блока всегда выделяются жирным;
- пустые подпункты скрываются;
- если полезных сигналов нет, это пишется явно.

Смысл блока:

- показать, где РОПу нужно вмешаться лично;
- оставить только короткий список приоритетов и действий на день;
- показать, где сегодня можно быстро взять деньги;
- показать, где процесс продажи сломан между обязательными встречами;
- не превращать блок в повтор полной операционной детализации.

## Принципы качества отчёта

- коротко и по делу;
- формат, удобный для Telegram;
- без воды;
- без дублей между соседними блоками;
- без нулевых менеджеров в блоке просрочек;
- без домыслов при отсутствии данных;
- с явной пометкой ограничения, если источник не подтверждён.

## Чего пока может не хватать

- рабочее подключение `Bitrix24` в текущем контуре не подтверждено;
- полная карта полей `Bitrix24` не подтверждена;
- доступ к задачам, коммуникациям и просрочкам не подтверждён;
- боевой `cron`, привязки и маршрутизация Telegram для этой роли ещё не подтверждены технически, хотя расписание уже зафиксировано;
- исполняемый runtime роли закреплён в каноническом репозитории через `agents/lev_petrovich`;
- legacy-имя `agents/sales_agent` сохранено только как compatibility-path на переходный период.
